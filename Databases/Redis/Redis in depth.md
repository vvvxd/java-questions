Инициализация конфигурации

Её можно грубо разделить на два этапа. Первый — инициализация значений конфигурации по умолчанию, жестко закодированных в исходном коде Redis. Эта логика реализована в процедуре `initServerConfig()`. Второй — чтение файла конфигурации и перезапись настроек по умолчанию; вы можете найти её в процедуре `loadServerConfig()`.

`initServerConfig()`

В файле `config.c` определена переменная `standardConfig static_configs[]`. Каждая структура `standardConfig` имеет, помимо прочего, значение по умолчанию для определяемой ею настройки и адрес поля `redisServer`, которое хранит это значение. Например, один из элементов `static_config[]` выглядит так:

```
createBoolConfig("appendonly", NULL, MODIFIABLE_CONFIG | DENY_LOADING_CONFIG, server.aof_enabled, 0, NULL, updateAppendonly)
```

Это означает, что значение по умолчанию для `server.aof_enabled` установлено в `0`. Значения конфигурации по умолчанию устанавливаются в процедуре `initConfigValues()`. Там для каждой структуры `standardConfig`, определенной в массиве `static_configs`, вызывается `config->interface.init(config)`. Поле `interface` представляет собой структуру `typeInterface`, которая в основном занимается получением и установкой данных определенного типа. Например, для булевой конфигурации процедура `init` устанавливается на `boolConfigInit` — это можно увидеть в макросе `createBoolConfig`, который инициализирует структуру `standardConfig`:

```
#define createBoolConfig(name, alias, flags, config_addr, default, is_valid, apply) { \
    embedCommonConfig(name, alias, flags) \
    embedConfigInterface(boolConfigInit, boolConfigSet, boolConfigGet, boolConfigRewrite, apply) \
    .type = BOOL_CONFIG, \
    .data.yesno = { \
        .config = &(config_addr), \
        .default_value = (default), \
        .is_valid_fn = (is_valid), \
    } \
}
```

Кроме того, каждый `standardConfig` добавляется в словарь, то есть в хеш-таблицу — чтобы по имени конфигурации можно было найти любой элемент за константное время. В Redis это представлено в виде структуры `dict`.

Более краткую версию в Miro можно посмотреть [здесь](https://www.google.com/search?q=...).

`loadServerConfig()`

Здесь каждая настройка, указанная в `redis.conf`, считывается и устанавливается с помощью процедуры `set` интерфейса `typeInterface`. Таким образом, значения по умолчанию перезаписываются.

Все процедуры манипуляции данными для каждой реализации `typeInterface` определены в `config.c`.

Создание прослушивающих сокетов

Совмещайте это описание с [визуальными материалами](https://www.google.com/search?q=...).

`connTypeInitialize()`

Всё начинается с инициализации `ConnectionType`. Эта структура содержит процедуры, связанные с подключением и специфичные для базового типа сокета. В ней есть процедуры как для клиента, так и для сервера. Среди прочего, они подключаются к конечной точке, создают прослушивающий сокет, привязанный к определенному адресу, принимают подключения, записывают данные в клиентский сокет, читают данные из прослушивающего сокета или сокета подключения, а также закрывают подключение.

Итак, на данный момент инициализация просто означает, что массив `static ConnectionType *connTypes[CONN_TYPE_MAX]` заполняется типами подключений для TCP-сокетов, доменных сокетов Unix и, если Redis был скомпилирован с `BUILD_TLS=yes`, для tls (который в Redis можно использовать только с TCP-сокетами).

`initListeners()`

Здесь происходят две главные вещи. Во-первых, создаются прослушивающие сокеты, а во-вторых, их файловые дескрипторы добавляются в набор epoll. Давайте теперь рассмотрим всё по порядку.

**Общий жизненный цикл сетевого приложения** Когда клиенты отправляют запросы на подключение, они сначала создают сокет на своей стороне. После этого они отправляют запрос на подключение, который сводится к системному вызову `connect()`. Чтобы принимать запросы на подключение, серверная сторона должна иметь прослушивающие сокеты. У каждого сокета есть выделенная память для очереди, в которой временно хранятся входящие клиентские запросы на подключение:

Каждый системный вызов `accept()`, выполненный для прослушивающего сокета, извлекает первый запрос на подключение и создает новый подключенный сокет. Этот сокет будет использоваться для связи с конкретным клиентом, который сделал этот запрос на подключение. Этот недавно созданный сокет подключения будет обрабатывать следующую команду, которую отправит клиент. Вот как выглядит этот процесс:

Когда сервер или клиент хотят завершить свое подключение, один из них первым выполняет системный вызов `shutdown()`. Он отключает отправку или получение данных через указанный сокет, чтобы можно было обработать все ожидающие запросы. После этого сокет закрывается, что удаляет базовый файловый дескриптор и освобождает его ресурсы.

Итак, на данный момент прослушивающие сокеты созданы. В случае TCP эта логика реализована в процедуре `listenToPort()`. Непосредственно привязка и прослушивание осуществляются в процедуре `anetListen()`.

Стоит отметить еще две вещи. Во-первых, адреса, к которым привязаны сокеты, устанавливаются как переиспользуемые. Во-вторых, созданные сокеты делаются неблокирующими (non-blocking), но что это значит?

**Переиспользуемые адреса** Если адрес не является переиспользуемым, что является настройкой по умолчанию, только один сокет может быть к нему привязан. В противном случае к нему может быть привязано более одного адреса, и ОС автоматически распределяет входящие запросы между различными сокетами. Вот как выглядит этот процесс:

Это полезно, когда вы хотите быстро перезапустить сервер, но ваша операционная система считает, что адрес всё еще используется каким-то сокетом, что в случае с Redis не так: он привязывает адрес к одному сокету. Так что это способ избежать подобной ситуации. Адрес становится переиспользуемым в процедуре `anetSetReuseAddr()`.

**Неблокирующие сокеты** Когда вы читаете данные из обычного блокирующего сокета, а их в наличии нет, системный вызов `read()` зависнет — или заблокируется — пока данные не станут доступны. Это не активное ожидание (busy-wait), то есть ваш процессор не используется, но дальнейшее выполнение кода просто не происходит. В случае неблокирующих сокетов вместо этого возвращается специальный код: это означает, что прямо сейчас нет данных для чтения. Таким образом, ваш поток не зависает, но ваш код должен быть готов к такому сценарию.

То же самое относится и к системному вызову `write()`. Но как запись может быть заблокирована? Сначала давайте посмотрим, как выглядит запись в сокет.

Когда вы выполняете системный вызов `write()`, вы на самом деле не пишете в базовый файловый дескриптор. Это не похоже на запись в файл. Более того, файла вообще не существует. Вместо этого есть файловый дескриптор, который является просто записью в таблице файловых дескрипторов, поддерживаемой ядром для каждого процесса, и некоторая память ядра, связанная с ним, которая называется буфером. Я уже писал об этом ранее.

Именно туда ядро фактически записывает ваши данные. Что произойдет, если вы запишете сообщение, которое не помещается в буфер? Если это блокирующий сокет, ваш системный вызов `write()` зависнет. Когда ядро в конечном итоге освободит немного памяти, и ваше сообщение поместится в буфер, вызов `write()` всё равно зависнет. Он будет ждать, пока это сообщение не будет доставлено получателю. И поскольку в TCP нет верхней границы того, когда именно доставляется сообщение, ваше приложение может зависнуть на неопределенный срок.

Теперь, если я пишу в неблокирующий сокет и сообщение не помещается в буфер целиком, записывается только часть сообщения и возвращается специальный код. Ваше приложение должно быть готово возобновить запись, когда буфер будет готов. И это подводит нас к экземпляру epoll.

**Экземпляр epoll** Экземпляр epoll — это структура данных ядра Linux, которая состоит из двух списков: списка интересов (interest list) и списка готовности (ready list). Когда вы хотите узнать, что у какого-то файлового дескриптора есть данные для чтения или он готов к записи, вы можете добавить файловый дескриптор в список интересов, который также называется набором epoll. Экземпляр epoll асинхронно отслеживает состояние зарегистрированных файловых дескрипторов и добавляет их в список готовности, как только они становятся готовыми к операциям ввода-вывода. Вы можете получить список готовности, вызвав процедуру `epoll_wait()`.

Каждый элемент набора epoll называется событием epoll (epoll event), что может звучать обманчиво. Обычно, когда происходит "событие", оно уже свершилось. Вы можете подписаться на события; через некоторое время они происходят, и вы можете каким-то образом получить это событие. Это же происходит и в случае с набором epoll, но `epoll_event` — вероятно, неудачное название для элемента списка интересов.

Вот что вы делаете, когда подписываетесь на события чтения или записи для ваших файловых дескрипторов. Сначала вы создаете структуру `epoll_event`. Затем вы указываете, что именно вас интересует. Если вы подписываетесь на события чтения, это будет выглядеть как `epoll_event.events |= EPOLLIN`. И, наконец, вы указываете интересующий вас файловый дескриптор: `epoll_even.data.fd = my_fd;`.

_Сокет с файловым дескриптором 1 имеет данные для чтения_

В Redis логика epoll реализована в файле `ae_epoll.c`. Чтобы получить события, которые фактически произошли с момента вашей последней проверки, вы можете вызвать `epoll_wait`.

**Обработчики событий (Event handlers)** Как Redis узнает, какую процедуру запустить при возникновении определенного события? Он хранит обработчики событий в массиве `aeFileEvent *events` структуры `aeEventLoop`. Его название кажется немного вводящим в заблуждение; думаю, следовало назвать его `handlers`. Индексы массива являются файловыми дескрипторами, поэтому Redis может быстро найти нужный обработчик. Если вы подписываетесь на события чтения, вы должны установить обработчик чтения, который хранится в поле `aeFileProc *rfileProc`, а если на события записи, то вы устанавливаете обработчик в `aeFileProc *wfileProc`.

`aeCreateFileEvent` — это процедура, которая подписывается на определенные события и устанавливает соответствующие обработчики.

**Краткое резюме** Итак, вкратце, вы подписываетесь на события чтения или записи для ваших файловых дескрипторов, а затем проверяете их состояние. Если вы делаете это периодически в бесконечном цикле, весь этот механизм называется циклом событий (event loop). И это именно то, что происходит с прослушивающими сокетами. Redis помещает их в набор epoll, подписывается на события чтения и устанавливает поле `aeFileProc *rfileProc` на `ConnectionType.accept_handler`, которое является `connSocketAcceptHandler` в случае TCP-подключения. После этого Redis периодически проверяет, есть ли у этих дескрипторов сокетов ожидающие клиентские подключения. Когда они есть, Redis их принимает. Об этом будет отдельный пост. Пока что вы можете посмотреть [эту схему](https://www.google.com/search?q=...).

Инициализация фонового ввода-вывода

Существуют некоторые системные вызовы, которые могут блокироваться на довольно продолжительное время. В первую очередь, это `fsync`, который фактически перемещает данные со страниц памяти ОС на устройство хранения. Кроме того, если большой объем памяти был выделен с помощью `malloc()` и для него вызывается `free()`, операционной системе может потребоваться выполнить некоторые закулисные операции, такие как дефрагментация памяти, что может занять значительное время и привести к блокировке вызова `free()`.

Именно поэтому Redis не запускает операции, использующие эти вызовы, в главном потоке. Вместо этого он использует специальные фоновые потоки — один поток на тип задачи, всего три потока. В исходном коде они часто называются биопотоками (bio threads), где "bio" означает фоновый ввод-вывод (**b**ackground **I**/**O**), хотя функция `free` не имеет ничего общего с вводом-выводом. Существует четыре типа задач:

```
enum {
    BIO_CLOSE_FILE = 0, /* Отложенный системный вызов close(2). */
    BIO_AOF_FSYNC,      /* Отложенный fsync для AOF. */
    BIO_LAZY_FREE,      /* Отложенное освобождение объектов. */
    BIO_CLOSE_AOF,      /* Отложенное закрытие файлов AOF. */
    BIO_NUM_OPS
};
```

Закрытие файла обычно не является блокирующим вызовом, и в большинстве случаев оно возвращается немедленно. Но бывают некоторые неочевидные случаи, когда оно может заблокироваться, и, что хуже, это поведение различается в разных системах. Поэтому в целом закрытие файловых дескрипторов в фоновом режиме является хорошей практикой. Более того, в Redis это делается с предварительным вызовом `fsync`, что является относительно дорогостоящим.

Каждый поток выполняет процедуру `bioProcessBackgroundJobs`. Если указана настройка `bio_cpulist`, она применяется здесь с помощью процедуры `redisSetCpuAffinity`. Это сводится к системному вызову `sched_setaffinity`. Фактически вы указываете набор процессоров для выполнения этих потоков. Это может снизить накладные расходы на переключение контекста, аннулирование кеша и другие задачи, связанные с ЦП, что приводит к сокращению времени выполнения и повышению производительности.

Затем бесконечный цикл обрабатывает список задач, специфичный для данного потока. Если он пуст, поток ждет сигнала, уведомляющего о новой задаче, и зависает:

```
if (listLength(bio_jobs[worker]) == 0) {
    pthread_cond_wait(&bio_newjob_cond[worker], &bio_mutex[worker]);
    continue;
}
```

Если же список не пуст, Redis берет первый элемент из списка `bio_jobs[worker]` и обрабатывает его.

`bioSubmitJob` — это процедура, используемая для добавления задачи для фонового выполнения.

Потоковый ввод-вывод

Обычно утверждают, что Redis является однопоточным. Это не означает, что в Redis есть только один главный поток для всего. Как вы уже видели в предыдущей части, существуют, например, фоновые потоки. Более того, Redis иногда порождает дочерние процессы (forks child processes). Итак, на самом деле это означает, что все клиентские команды обрабатываются одним главным потоком на уровне хранения данных.

Например, если у вас есть десять тысяч одновременно подключенных клиентов, и все они одновременно отправляют команду `SET` для одного и того же ключа; это означает, что Redis установит этот ключ с указанным значением десять тысяч раз, один за другим, по одному разу, в полной изоляции друг от друга, в единственном главном потоке.

Но этот факт, тем не менее, оставляет некоторое пространство для оптимизации в конкретных сценариях. Основной целевой сценарий выглядит следующим образом. Если у вас одновременно подключено множество клиентов, и они отправляют команды, которые имеют большие ответы, может потребоваться некоторое время, чтобы перебрать всех этих клиентов и для каждого из них выполнить системный вызов `write()`.

Эта команда копирует данные из Redis, которые находятся в пользовательском пространстве, в буфер TCP, который находится в пространстве ядра. Итак, во-первых, это изначально медленно из-за пересечения этой границы, а во-вторых, чем больше ответ, тем больше времени требуется для его копирования. И Redis придумал решение, нацеленное на эту проблему: он может записывать ответы в потоках, работающих параллельно. Эти потоки обычно называются потоками ввода-вывода (io threads). Количество потоков можно установить с помощью настройки `io-threads`.

Второй сценарий концептуально такой же, но нацелен на парсинг команд. Поскольку большие тела запросов от клиентов встречаются реже, этот случай еще более редкий. Это поведение управляется отдельной настройкой `io-threads-do-reads`.

Реализация потоков ввода-вывода более подробно рассмотрена на [этой схеме](https://www.google.com/search?q=...). Потоки запускаются в процедуре `initThreadedIO()`. `IOThreadMain` — это процедура, которая выполняется в каждом потоке.

Пара заметок о реализации: Во-первых, как Redis понимает, что есть какие-то ответы для записи или какие-то входящие запросы для парсинга? Он проверяет процедуру `getIOPendingCount`, которая возвращает количество элементов для обработки потоком, чей `id` передан в качестве аргумента. Во-вторых, как Redis знает, что именно ему следует сделать — отправить ответ или распарсить команду? Это определяет глобальная статическая переменная `io_threads_op`. Вот как использовать эту переменную:

```
if (io_threads_op == IO_THREADS_OP_WRITE) {
    writeToClient(c,0);
} else if (io_threads_op == IO_THREADS_OP_READ) {
    readQueryFromClient(c->conn);
}
```

В случае записи ответов клиентам как операция потока, так и количество ожидающих ответов устанавливаются синхронно в `handleClientsWithPendingWritesUsingThreads`. Чтение выполняется в `handleClientsWithPendingReadsUsingThreads`.

И последнее, но не менее важное: может показаться, что потоки ввода-вывода находятся в состоянии активного ожидания (busy-waiting), когда им нечего делать. Вот код:

```
for (int j = 0; j < 1000000; j++) {
    if (getIOPendingCount(id) != 0) break;
}
/* Дать главному потоку шанс остановить этот поток. */
if (getIOPendingCount(id) == 0) {
    pthread_mutex_lock(&io_threads_mutex[id]);
    pthread_mutex_unlock(&io_threads_mutex[id]);
    continue;
}
```

На самом деле это не так. Их состояние контролируется блокировкой соответствующих мьютексов `io_threads_mutex[id]` — вы можете видеть это в комментарии, который гласит `Give the main thread a chance to stop this thread`. Изначально они заблокированы главным потоком. Когда Redis решает, что клиентов достаточно, чтобы оправдать их обработку потоками ввода-вывода, он разблокирует их мьютексы в процедуре `startThreadedIO()`. А когда количество активных клиентов уменьшается, Redis просто снова захватывает блокировку в процедуре `stopThreadedIO()`. Это приводит к зависанию потоков ввода-вывода, когда они пытаются заблокировать уже заблокированный мьютекс.

Я напишу отдельный пост о том, как вся обработка клиентских запросов осуществляется потоками ввода-вывода. А пока вы можете перейти к [этой схеме в Miro](https://www.google.com/search?q=...).

Загрузка клиентских данных с диска в память

Сначала краткий обзор того, как выглядит персистентность (сохранение данных) в Redis.

Два варианта персистентности в Redis

Первый — это создание снимка данных, всех данных экземпляра Redis в одном бинарном файле. Этот файл называется файлом RDB, и весь этот механизм обычно называется RDB, что означает базу данных Redis (**R**edis **d**ata**b**ase). Имя файла можно настроить с помощью параметра `dbfilename`.

Второй — это логирование всех мутирующих команд в удобочитаемом формате — том самом, который используется в протоколе Redis. Все файлы, создаваемые Redis при использовании этой опции, собирательно называются файлами AOF, что означает файл только для добавления (**a**ppend-**o**nly **f**ile). Вы можете включить эту опцию с помощью настройки `appendonly`. Конкретная реализация базового механизма меняется в разных версиях Redis.

**AOF в Redis 7**

В Redis 7 механизм AOF был значительно переработан. Он использует два типа файлов для хранения клиентских данных.

Первый тип — это базовый файл (base file). Это снимок набора данных всего экземпляра Redis. Его формат зависит от настройки `aof-use-rdb-preamble`. Если она включена, базовый файл содержит необработанные бинарные данные. В этом случае базовый файл ничем не отличается от файла RDB. Если она выключена, базовый файл содержит данные в виде сжатого списка команд.

Например, если какой-то клиент сто раз устанавливал для определенного ключа разные значения, то в базовый файл попадет только последняя фактическая команда. То же самое относится и к другим структурам данных. Если клиент добавил сто значений в список по одному элементу на команду, базовый файл будет содержать одну команду `RPUSH` со ста аргументами.

Второй тип — это инкрементный файл, или, для краткости, файл `incr`. Именно сюда записываются все выполняемые в данный момент мутирующие команды. Они хранятся "как есть".

Когда сервер Redis запускается, он проверяет конфигурацию на наличие используемого в данный момент механизма персистентности и загружает соответствующий файл. В случае с AOF он сначала загружает базовый файл, а затем файл `incr`.

По мере записи новых данных в хранилище текущему файлу `incr` требуется ротация, чтобы он не рос бесконечно. Этот процесс ротации, обычно называемый перезаписью AOF (AOF rewrite), выполняется периодически. Вместо того чтобы перезаписывать существующие файлы, создаются новые. Вкратце, это происходит так. Сначала создается новый файл `incr`. Теперь все текущие команды записываются в этот новый файл. После этого делается новый снимок. Когда он будет завершен, предыдущие версии базового файла и файла `incr` будут удалены. Я описываю этот процесс более подробно [здесь](https://www.google.com/search?q=...), и я напишу об этом чуть позже.

Теперь давайте рассмотрим процесс загрузки. Как обычно, схему вы можете найти [здесь](https://www.google.com/search?q=...).

Реализация загрузки файлов AOF в память

Как я писал ранее, данные клиентов хранятся в двух типах файлов. У них может быть несколько версий одновременно, поэтому их нужно как-то отслеживать. Для этого и нужен файл манифеста (manifest file). Он хранит текущие активные версии файла `base` и файлов `incr`, а также их предыдущие, исторические (historical) версии, которые вскоре будут удалены. Если настройка `appendfilename` равна `my_appendonly_file`, файл манифеста может выглядеть так:

```
# это базовый файл, тип "b" от "base" 
file my_appendonly_file.aof.2.base.aof seq 2 type b

# Это предыдущая версия базового файла, обратите внимание на префикс ".base" в имени файла.
# Тип h означает "historical" (исторический)
file my_appendonly_file.aof.1.base.aof seq 1 type h

# это предыдущая версия файла incr. Обратите внимание на префикс ".incr".
file my_appendonly_file.aof.1.incr.aof seq 1 type h

# это текущий активный файл incr. Все текущие команды записываются сюда
file my_appendonly_file.aof.2.incr.aof seq 2 type i
```

Процесс перезаписи AOF начинается с загрузки этого файла манифеста в память, а именно в структуру `aofManifest`. Вот как она выглядит:

```
typedef struct {
  aofInfo     *base_aof_info;     /* Информация о файле BASE. NULL, если файла BASE нет. */
  list        *incr_aof_list;     /* Список файлов INCR AOF. Может быть несколько INCR AOF, если перезапись не удалась. */
  list        *history_aof_list;  /* Список HISTORY AOF. При успешной AOFRW информация из `base_aof_info` и `incr_aof_list` будет перемещена в этот список. Мы удалим эти файлы AOF после завершения AOFRW. */
  long long   curr_base_file_seq; /* Порядковый номер, используемый текущим файлом BASE. */
  long long   curr_incr_file_seq; /* Порядковый номер, используемый текущим файлом INCR. */
  int         dirty;              /* 1 указывает на то, что aofManifest в памяти несовместим с диском, нам нужно немедленно его сохранить. */
} aofManifest;
```

Это делается в процедуре `aofLoadManifestFromDisk()`.

Затем загружаются данные клиента. Сначала Redis проверяет, какой механизм персистентности используется в данный момент. Если AOF отключен, то загружается снимок RDB. В противном случае загружаются файлы AOF: сначала базовый файл, а затем все файлы `incr`. Обычно существует только один файл `incr`. Но если предыдущий процесс перезаписи AOF завершился с ошибкой, в Redis может быть более одного файла `incr`. Входящие команды записываются в один единственный файл `incr` в любой момент времени.

> Важное следствие: если вы использовали только механизм RDB, когда вы останавливаете Redis, затем включаете AOF и затем запускаете свой экземпляр, ваши данные из файла RDB загружены не будут. Как и почти любую настройку в Redis, вы можете переключить AOF "на лету": `CONFIG SET appendonly yes`.

`loadSingleAppendOnlyFile()` — это процедура, которая загружает файл AOF. В ней есть проверка на обратную совместимость, поэтому эта процедура может работать с файлами, созданными в предыдущих версиях Redis. Раньше, если был включен параметр `aof-use-rdb-preamble`, файл AOF содержал так называемую преамбулу. Это был просто начальный набор данных Redis в бинарном формате, ничем не отличающийся от снимка RDB. Поэтому, если Redis запускается впервые после обновления, эта процедура сначала читает эту преамбулу, если она присутствует, а затем переходит к так называемому хвосту AOF. В Redis 7 базовый файл может содержать либо необработанные бинарные данные, либо сжатый список команд, но не то и другое вместе.

Когда данные загружены, Redis открывает файл incr из манифеста, чтобы он был готов для записи клиентских команд. Если это первый запуск Redis 7, то ни базового файла, ни файла `incr` не существует. В этом случае они создаются, а файл манифеста обновляется.

Наконец, если исторические файлы остались после последней перезаписи AOF, Redis удаляет их сейчас в процедуре `aofDelHistoryFiles()`. Как и любое удаление файлов в Redis, это делается асинхронно в биопотоке (bio thread).

Цикл событий (Event Loop)

Обзор

Я уже упоминал концепцию цикла событий, когда рассказывал о наборе epoll. Но механизм epoll — это деталь реализации цикла событий, который сам по себе является просто абстрактной концепцией. Это программная конструкция, которая позволяет программе эффективно обрабатывать множественные операции ввода-вывода без блокировки или пустой траты циклов процессора. Типичные случаи — это сетевой ввод-вывод и файловый ввод-вывод. То, что приводит в действие любой цикл событий, является специфическим механизмом мультиплексирования. Конкретная реализация ядра отслеживает зарегистрированные файловые дескрипторы и сообщает об их состоянии по вашему запросу. Epoll — одна из таких реализаций, самая последняя и самая производительная. Сейчас она используется в Redis, если он собирается в системах Linux.

Независимо от механизма реализации, цикл событий состоит из следующих базовых шагов:

1. Зарегистрировать интерес в событиях ввода-вывода для каждого файлового дескриптора, используя соответствующий системный вызов. Другими словами, вы подписываетесь на событие. Событие ввода-вывода — это, по сути, изменение состояния файлового дескриптора. Например, если у вас есть прослушивающий сокет и какой-то клиент подключается к адресу, привязанному к нему, состояние этого сокета меняется: теперь в его буфере TCP есть запрос на подключение, ожидающий чтения и принятия.

2. Ожидать событий ввода-вывода на зарегистрированных файловых дескрипторах, заблокировав системный вызов в зависимости от вашего механизма мультиплексирования. Вы можете ждать бесконечно, пока не произойдет хотя бы одно событие, или с определенным тайм-аутом.

3. Когда блокирующий системный вызов из предыдущего шага возвращается, вы получаете события, которые произошли на данный момент.

4. Определить тип произошедшего события (чтение, запись и т. д.) и обработать его должным образом, выполнив соответствующую функцию обратного вызова (callback function).

5. Повторить.


Цикл событий с механизмом epoll

Сначала Redis создает так называемый экземпляр epoll: `epoll_create(1024);`. Этот системный вызов фактически создает контейнер для набора epoll и списка готовности и возвращает связанный с ним файловый дескриптор. В ранних версиях Linux его единственный аргумент подсказывал ядру, какого размера следует ожидать набор epoll. Но сейчас он игнорируется. Чтобы начать отслеживать состояние файлового дескриптора, вы вызываете системный вызов `epoll_ctl`. Помимо добавления элемента в набор epoll, эта процедура также может изменять и удалять его. А когда вы хотите получить все события, произошедшие с зарегистрированными файловыми дескрипторами, вы вызываете `epoll_wait`.

Цикл событий в Redis

Я собирал, устанавливал и запускал Redis в Linux, поэтому я буду ссылаться на epoll как на механизм мультиплексирования. Как обычно, вы можете сочетать чтение этого текста с [этой схемой](https://www.google.com/search?q=...).

Во-первых, экземпляр epoll создается в процедуре `aeApiCreate`, которая вызывается во время инициализации цикла событий в `aeCreateEventLoop`. Чтобы отследить состояние какого-либо файлового дескриптора, вы можете вызвать `aeApiAddEvent()`. `aeApiDelEvent()` удаляет файловый дескриптор из списка интересов.

В качестве последнего шага при запуске сервера Redis запускает бесконечный цикл процедуры `aeProcessEvents`, фактически реализуя всю итерацию цикла событий. Она состоит из нескольких логических частей.

1. Первая — это вся логика, которая должна быть выполнена перед ожиданием событий. Поскольку это ожидание блокируется с определенным тайм-аутом, логика, выполняемая перед `epoll_wait`, собрана в процедуре под названием `beforesleep`. Она реализована здесь. Среди прочего, она вызывает процедуры, занимающиеся чтением команд клиентов и записью им ответа. Я напишу отдельный пост о нескольких сценариях, когда клиент отправляет запрос, а Redis его обрабатывает. Тем не менее, вы уже можете найти соответствующие схемы [здесь](https://www.google.com/search?q=...).

2. После этого Redis запрашивает все события, которые произошли с момента последней итерации цикла событий. Эта логика реализована в процедуре `aeApiPoll`. В ней Redis вызывает системный вызов `epoll_wait`. Он всегда делается с определенным тайм-аутом, поэтому никогда не блокируется на неопределенное время. Затем Redis собирает сработавшие события в `aeEventLoop->fired`.

3. Затем Redis проходит по `aeEventLoop->fired` в цикле и запускает соответствующие обработчики. Они хранятся в массиве `aeEventLoop->events`. Обработчики добавляются путем вызова процедуры `aeCreateFileEvent`.

4. Наконец, обрабатываются так называемые временные события (time events). Это события, указывающие на то, что прошел определенный временной интервал. В C они известны как события таймера. Вы можете подписаться на них штатным образом, создав таймер с помощью `timerfd_create` и поместив полученный файловый дескриптор в набор epoll, а затем установив тайм-аут с помощью процедуры `timerfd_settime`. Это может пригодиться для выполнения логики, которую необходимо запускать периодически. Однако в Redis на самом деле нет такой вещи, как классическое событие таймера. Вместо этого он эмулирует их. Сначала вы регистрируете обработчик с помощью процедуры `aeCreateTimeEvent`, где вы фактически указываете временную метку (timestamp), когда должен сработать ваш обработчик. Затем, в конце каждой итерации цикла событий, вызывается `processTimeEvents`. Там Redis итерирует каждый обработчик и проверяет, наступила ли указанная временная метка. Если наступила, выполняется соответствующий обработчик. Именно так, например, реализован cron в Redis. Вскоре мы увидим, что там происходит, но сначала краткий обзор процедуры `beforeSleep()`.

`beforeSleep()`

Существуют некоторые процедуры, которые должны быть выполнены до получения новых событий. Как я уже писал, среди них есть процедуры, работающие с вводом и выводом клиентов; об этом будет отдельный пост. Здесь я кратко пройдусь по процедурам, связанным с блокировкой клиентов. Надеюсь, когда-нибудь я обновлю эту часть функцией отслеживания инвалидации ключей, которая использует механизм pub/sub.

Блокировка клиентов

Существует набор процедур, выполняемых в `beforeSleep()`, которые все относятся к сценарию, когда клиентское соединение блокируется. Блокировка означает, что Redis не парсит и, следовательно, не выполняет никакие команды, которые отправляет клиент, хотя он читает и сохраняет эти необработанные запросы в буфере клиента.

**Обзор** Во-первых, давайте рассмотрим, как вообще работает эта блокировка на примере команды `BRPOP`.

Когда клиент отправляет команду `BRPOP`, Redis блокирует клиента для указанных ключей с заданным тайм-аутом, если ни один из этих ключей не существует. Эта логика реализована в процедуре `blockForKeys()`. В ней каждый клиент сначала устанавливает ключи, на которых он блокируется. Затем каждый ключ добавляется в словарь `redisDb->blocking_keys`, а его значением является список клиентов, заблокированных на нем. Таким образом, Redis имеет доступ как к ключам, заблокированным данным клиентом, так и к клиентам, заблокированным на данном ключе. Кроме того, тайм-аут обрабатывается аналогичным образом. Он устанавливается для текущего заблокированного клиента, а также добавляется в radix tree `server.clients_timeout_table`. И, наконец, клиент помечается как заблокированный.

Когда какой-то другой клиент добавляет первый элемент в любой из несуществующих списков, на которых блокируется первый клиент, этот ключ добавляется в базу данных. Процедура `dbAdd()` проверяет, есть ли какие-либо клиенты, заблокированные на нём, и если есть, этот ключ добавляется в список `server.ready_keys`.

**`handleBlockedClientsTimeout()`** Эта процедура проходит по radix tree `server.clients_timeout_table` и получает всех клиентов с истекшим тайм-аутом. Каждый из них разблокируется. Это означает, что клиент больше не помечен как заблокированный, поэтому с этого момента все его дальнейшие команды будут обрабатываться. Наконец, Redis помещает его в список `server.unblocked_clients`, который будет обработан в следующей части.

**`processUnblockedClients()`** Здесь Redis итерирует список `server.unblocked_clients`. Как я писал выше, в нем находятся все клиенты, разблокированные недавно. Если любой из них отправлял какие-либо команды во время блокировки, они были приостановлены; теперь пришло время их распарсить и выполнить. Я напишу отдельный пост о сценарии выполнения команд, но вы можете [посмотреть схему Miro](https://www.google.com/search?q=...) прямо сейчас.

**`handleClientsBlockedOnKeys()`** Эта процедура перебирает недавно добавленные клиентские ключи, на которых блокируется как минимум один клиент. Как я упоминал ранее, они хранятся в списке `server.ready_keys`. Для каждого такого ключа каждый клиент, который на нём блокируется, разблокируется. Как я уже писал, с этого момента все его дальнейшие команды будут обрабатываться: как парситься, так и выполняться. Кроме того, он попадает в список `server.unblocked_clients`, который Redis итерирует в процедуре `processUnblockedClients()`, описанной выше.

Redis Cron

Redis cron относится к процедуре `serverCron`. Она запускается `hz` раз в секунду. Но если параметр `dynamic-hz` равен 1 (так по умолчанию), то частота cron адаптируется в зависимости от количества активных клиентов:

```
if (server.dynamic_hz) {
    while (listLength(server.clients) / server.hz >
           MAX_CLIENTS_PER_CLOCK_TICK)
    {
        server.hz *= 2;
        if (server.hz > CONFIG_MAX_HZ) {
            server.hz = CONFIG_MAX_HZ;
            break;
        }
    }
}
```

Тут происходит много всего. Обновляется статистика Redis, происходит корректное завершение работы, некоторая информация пишется в логи, выполняется некоторая отложенная логика — я не буду в это углубляться. Я опишу то, что считаю наиболее важными частями.

`clientsCron()`

На самом деле, раздел комментариев этой процедуры уже отлично описывает то, что здесь происходит. Вкратце, она берет всех активных клиентов и обрабатывает `1 / server.hz` из них, так что все клиенты должны быть обработаны примерно за одну секунду. Для каждого клиента Redis выполняет следующие процедуры:

1. Если указан тайм-аут клиента, Redis рассчитывает количество времени, в течение которого текущий клиент бездействовал. Если оно превышает значение настройки, клиент удаляется и соединение закрывается — файловый дескриптор сокета соединения сначала удаляется из цикла событий, а затем закрывается.

2. Размер поля `querybuf` структуры `client` уменьшается при необходимости. Именно здесь хранится тело клиентского запроса до того, как он будет распарсен. То же самое делается с выходным буфером клиента, который служит для хранения данных, ожидающих записи в сокет соединения.

3. Обновляется статистика использования памяти клиентами (`clientsCronTrackExpansiveClients` касается клиентов, _дорогих_ с точки зрения используемой ими памяти).

4. Если достигнут `client-output-buffer-limit`, клиент немедленно закрывается.


Исторически сложилось так, что Redis использовал здесь следующий трюк. В начале каждой итерации хвост `server.clients` перемещается в голову. Зачем? Если этот клиент удален, избегается сложность `O(n)`: Redis начинает поиск элемента с начала списка, но поскольку он всегда первый, поиск всегда занимает константное время. Однако, насколько я понимаю, в этом больше нет необходимости. Когда запрос на клиентское соединение принимается и создается структура `client`, элемент списка `server.clients` кэшируется там:

```
void linkClient(client *c) {
    listAddNodeTail(server.clients,c);
    /* Обратите внимание, что мы запоминаем узел связанного списка, в котором хранится клиент,
     * таким образом удаление клиента в unlinkClient() не потребует
     * линейного сканирования, а только операции за константное время. */
    c->client_list_node = listLast(server.clients);
```

Более того, поскольку новые клиенты добавляются в хвост списка `server.clients`, `clientsCron()` обработает их первыми, оставляя старых клиентов без внимания на некоторое время.

`databasesCron()`

Здесь находятся процедуры, необходимые для обслуживания уровня хранилища Redis.

**`activeExpireCycle(ACTIVE_EXPIRE_CYCLE_SLOW)`** Вы можете найти [визуальный аналог здесь](https://www.google.com/search?q=...).

Эта процедура удаляет просроченные ключи. Они также удаляются при извлечении, но что, если они никогда не извлекаются? Они могут заполнить всю доступную память. Вот почему срок действия ключей истекает проактивно.

Ключи с заданным временем истечения хранятся в поле `expires` структуры `redisDb`, которое представляет собой структуру `dict`. В каждом `dict` на самом деле есть две хеш-таблицы для постепенного изменения размера. Это необходимо, когда количество элементов в хеш-таблице становится слишком большим, поэтому размер хеш-таблицы должен быть скорректирован соответствующим образом. Или же количество элементов может стать слишком маленьким, поэтому хеш-таблица сжимается для большей эффективности. Я напишу об этом больше позже. А пока достаточно того, что их две и обе они реализованы как массивы.

**`activeDefragCycle()`** Дефрагментация — это процесс уменьшения фрагментации памяти, которая возникает, когда свободная память в программе делится на мелкие фрагменты, слишком маленькие, чтобы быть полезными для выделения новых объектов или данных. Это может привести к неэффективному использованию памяти и снижению производительности программы с течением времени. Дефрагментация помогает объединить свободную память в более крупные непрерывные блоки, которые затем можно более эффективно использовать для новых выделений. Этот процесс довольно дорогостоящий и по умолчанию в Redis отключен. Обычно вы можете определить, что вам нужна дефрагментация, периодически проверяя вывод `info memory`.

Сама дефрагментация сводится к перераспределению памяти для текущего указателя в случае, если это целесообразно. Redis выполняет этот процесс инкрементно, так же как он подходит к истечению срока действия ключей. То есть он не выполняет всю необходимую работу за один прогон, потому что это может занять некоторое время, и в эти периоды Redis не будет отвечать на запросы. Вместо этого такие операции выполняются небольшими шагами, контролируемым образом с жесткими ограничениями, но часто. Он перебирает хеш-таблицы с помощью процедуры `dictScanDefrag()`, проходит по корзинам хеш-таблицы и для каждой корзины дефрагментирует три типа указателей: `dictEntry`, его `key` и `v.val`. Он начинает с `dictEntry`:

1. Сначала для текущей корзины вызывается `dictDefragBucket()`. Там `activeDefragAlloc()` выполняется для каждой записи корзины. Это сводится к двум вещам. Во-первых, Redis спрашивает аллокатор памяти jemalloc, стоит ли перераспределять текущий указатель. И если да, память этого указателя перераспределяется, а память старого указателя копируется в новый. После этого старый указатель освобождается.

2. После этого Redis снова проходит по каждому `dictEntry` в текущей корзине и для каждого выполняет `defragScanCallback()`. Эта процедура сводится к вызову процедуры `defragKey()`. Сначала он дефрагментирует ключ. Затем Redis выполняет дефрагментацию значения, используя процедуру, которая зависит от типа текущего `dictEntry.v.val`.


**`tryResizeHashTables()`** Если количество элементов в хеш-таблице становится слишком малым по сравнению с её размером, это может привести к нескольким проблемам. Во-первых, большое количество корзин означает, что для хеш-таблицы выделено много памяти, но большая часть её остается неиспользованной. Во-вторых, производительность снижается при сканировании разреженной хеш-таблицы. Вот почему Redis периодически уменьшает недоиспользуемые хеш-таблицы. Процедура `tryResizeHashTables()` делает именно это: она уменьшает размер хеш-таблиц, если они заполнены менее чем на десять процентов.

Процесс изменения размера никогда не затрагивает хеш-таблицу с клиентскими данными. Представьте себе хеш-таблицу с миллионом ключей. Если ее размер изменяется на месте, каждый ключ нужно будет перехешировать и переместить в новый слот. Это может занять некоторое время. Вместо этого в `dict->ht_table[1]` выделяется хеш-таблица нового размера:

Её размер устанавливается равным ближайшей степени двойки, большей или равной количеству элементов в хеш-таблице.

Изменение размера здесь только сужает словарь. Расширение происходит в нескольких местах, в первую очередь — при вставке нового значения, а именно в процедуре `dictFindPositionForInsert()`.

Обратите внимание, что создание новой хеш-таблицы и перенос туда данных из старой (то есть `dict->ht_table[0]`) — это два разных процессов. Процесс перемещения данных в новую хеш-таблицу называется перехешированием (rehashing). Это делается инкрементно при взаимодействии со словарем, как в операциях чтения, так и в операциях записи. Например, посмотрите `dictFind()`, `dictGetRandomKey()`, `dictFindPositionForInsert()` и т.д. Кроме того, это выполняется в каждой итерации cron с помощью процедуры `incrementallyRehash()`.

**Порождение нового процесса (forking) и метод copy-on-write** Это примерно всё, что происходит в процедуре `databasesCron()`. Следует отметить, что все эти манипуляции с данными в памяти разрешены только тогда, когда нет дочернего процесса. Зачем? Когда порождается дочерний процесс, он получает доступ ко всем данным, которые есть у родительского процесса. Это происходит благодаря методу копирования при записи (copy-on-write, CoW), реализованному ОС. Но если какой-либо из процессов изменяет общие данные, базовая страница памяти копируется и становится видимой только для того процесса, который произвёл изменение. Другой процесс по-прежнему видит данные, которые были актуальны на момент порождения.

Эта технология CoW имеет сопутствующие затраты, такие как повышенное использование пространства и потенциальные издержки производительности. Именно поэтому Redis избегает задач обслуживания, когда запущен дочерний процесс. По этой причине вы периодически встречаете такую проверку по всей кодовой базе. Кроме того, мутация данных в дочернем процессе бесполезна — родитель её всё равно не увидит. Вот почему изменение размера — и, следовательно, последующее перехеширование — отключены в дочернем процессе.

`rdbSaveBackground()`

Эта процедура создает снимок rdb. По умолчанию эта процедура запускается раз в час, если есть хотя бы одно изменение, каждые 5 минут, если есть как минимум 100 изменений, и раз в минуту при как минимум 10000 изменений. Эти настройки можно изменить.

Вся тяжелая работа происходит в дочернем процессе. Там выполняется процедура `rdbSave()`, и процесс завершается. Её реализация довольно прямолинейна, но утомительна. Все данные экземпляра сначала записываются в файл. Из соображений производительности некоторые из этих данных могут храниться в буфере внутри подсистемы ввода-вывода библиотеки C. Поэтому, когда запись завершена, Redis сначала должен убедиться, что все эти данные перемещены в указанный файл. Именно это делает процедура `fflush`. Однако это не гарантирует, что данные были полностью записаны на диск. Они могут быть закэшированы ОС или контроллером диска. Чтобы фактически переместить эти данные на диск, необходим вызов `fsync`.

Процедура `rdbSaveRio()` в Redis отвечает за запись содержимого базы данных Redis на диск. Она начинает с записи некоторых метаданных и определений функций, а затем переходит к записи содержимого каждой базы данных. Для каждой записи в базе данных процедура записывает ключ и значение. Ключу предшествует идентификатор типа значения, а само значение записывается в его необработанном бинарном виде. В дополнение к ключу и значению процедура записывает время истечения срока действия ключа, а также информацию об удалении (eviction) LRU или LFU, в зависимости от настроенной политики удаления.

`rewriteAppendOnlyFileBackground()`

Я уже писал о процессе загрузки данных из файлов в случае механизма AOF. Здесь описывается процесс перезаписи.

Он запускается, когда выполняются все следующие условия:

1. AOF включен (то есть параметр `appendonly` равен `1`)

2. процесс перезаписи еще не запущен

3. текущий размер файла AOF превышает параметр `auto-aof-rewrite-min-size`

4. размер файла AOF вырос больше чем на значение `auto-aof-rewrite-percentage` с момента последней перезаписи

5. и процесс перезаписи не отложен из-за повторяющихся сбоев


Этот процесс выполняется в несколько этапов.

**Сброс буфера Redis в файл AOF** Прежде всего, если в памяти Redis есть команды, еще не записанные в файл AOF, это делается немедленно.

**Создание нового файла incr, куда записываются текущие команды** Затем создается новый файл incr со следующим порядковым номером и добавляется в файл манифеста. Например, до начала процесса перезаписи файл манифеста выглядел так:

```
file my_appendonly_file.aof.2.base.aof seq 2 type b

 Если предыдущая перезапись aof завершилась ошибкой, здесь могут остаться файлы истории.
 Они удаляются в конце успешной перезаписи aof.
file my_appendonly_file.aof.1.base.aof seq 1 type h
file my_appendonly_file.aof.1.incr.aof seq 1 type h

 Это текущий активный файл incr.
 Все текущие команды логируются сюда.
file my_appendonly_file.aof.2.incr.aof seq 2 type i
```

Он может содержать файлы истории, если последняя перезапись завершилась сбоем. Затем предыдущий файл incr синхронизируется на диск (fsync) и закрывается. И, наконец, новый файл incr устанавливается для записи текущих команд. Теперь файл манифеста выглядит так:

```
file my_appendonly_file.aof.2.base.aof seq 2 type b

file my_appendonly_file.aof.1.base.aof seq 1 type h
file my_appendonly_file.aof.1.incr.aof seq 1 type h

file my_appendonly_file.aof.2.incr.aof seq 2 type i
file my_appendonly_file.aof.3.incr.aof seq 3 type i
```

**Снимок всего экземпляра Redis записывается во временный базовый файл** После того как новый файл incr готов для записи всех мутирующих команд, порождается новый дочерний процесс. Там и происходит фактическая перезапись. Она реализована в процедуре `rewriteAppendOnlyFile()`. Весь набор данных Redis записывается во временный файл под названием `temp-rewriteaof-bg-<current pid>.aof`. Если `aof-use-rdb-preamble` включен (так по умолчанию), то данные записываются в необработанном бинарном формате — в том же самом, который используется в снимке rdb. Если он отключен, данные записываются как удобочитаемый сжатый список команд. Он проходит по всем `dictEntry` в каждом `dict` и, в зависимости от типа значения, пишет команду set. Например, для ключа `key5`, имеющего простое строковое значение `value5`, он пишет:

```
*3
$3
SET
$4
key5
$6
value5
```

Это имеет форму команды, но в отличие от файла incr, в этом файле нет истории. Каждый ключ записывается только один раз, с его текущим значением.

Стоит отметить, что во время создания снимка родительский процесс продолжает обрабатывать команды клиентов, в том числе и мутирующие, которые Redis записывает в новый файл incr. Однако дочерний процесс, который делает снимок, этих изменений не видит. Он видит только снимок данных на момент порождения. В результате данные в снимке и актуальный набор данных ни пересекаются, ни имеют пропусков. Следовательно, при их загрузке в память во время запуска Redis не происходит дублирования или потери данных.

**Когда снимок готов, он становится новым базовым файлом** В каждой итерации cron Redis проверяет, завершено ли создание снимка. Если да, то временный файл переименовывается в новый базовый файл, а предыдущий базовый файл становится историческим. Кроме того, все файлы incr, за исключением текущего активного, также становятся историческими. Когда файл становится историческим, он не переименовывается; скорее, он помечается как таковой в файле манифеста, который в этот момент сохраняется на диск. Теперь он выглядит так:

```
# это новый базовый файл
file my_appendonly_file.aof.3.base.aof seq 3 type b

# это предыдущий файл incr
file my_appendonly_file.aof.2.incr.aof seq 2 type h
# это предыдущий базовый файл
file my_appendonly_file.aof.2.base.aof seq 2 type h
# это базовый файл, оставшийся от последней неудачной перезаписи aof
file my_appendonly_file.aof.1.base.aof seq 1 type h
# это файл incr, оставшийся от последней неудачной перезаписи aof
file my_appendonly_file.aof.1.incr.aof seq 1 type h

# это текущий активный файл incr, куда записываются входящие команды
file my_appendonly_file.aof.3.incr.aof seq 3 type i
```

Наконец, исторические файлы удаляются как с диска, так и из файла манифеста, который снова обновляется:

```
file my_appendonly_file.aof.3.base.aof seq 3 type b

file my_appendonly_file.aof.3.incr.aof seq 3 type i
```

## Краткое резюме запуска сервера Redis

Здесь я пройдусь по концепциям, на которые опирается процесс подключения клиента.

### Экземпляр epoll

Экземпляр epoll — это структура данных в ядре Linux, которая содержит список интересов (interest list) и список готовности (ready list). Когда файловый дескриптор добавляется в список интересов, также известный как набор epoll, экземпляр epoll асинхронно отслеживает его на наличие событий. События — это изменения в состоянии файлового дескриптора. Существует два основных типа таких изменений.

Первый — когда у файлового дескриптора появляются данные, доступные для чтения. Другими словами, он становится читаемым: его состояние меняется с «не готов» на «готов к чтению». Второй тип возникает, когда в буфере файлового дескриптора появляется свободное место для записи. В этом случае такие файловые дескрипторы обычно называют доступными для записи. Как только происходит любое из этих событий, экземпляр epoll добавляет файловый дескриптор в список готовности:

_Файловый дескриптор 1 имеет некоторые данные для чтения_

Чтобы получить файловые дескрипторы, готовые к операциям ввода-вывода, вы можете вызвать процедуру `epoll_wait()`. Epoll — это один из нескольких так называемых механизмов мультиплексирования, которые позволяют асинхронно отслеживать состояние файловых дескрипторов. Он используется в Redis, если он собран под Linux. Другие системы используют другие реализации этой же концепции.

### Цикл событий (Event loop)

Цикл событий — это программная конструкция, построенная поверх механизма мультиплексирования системы. По сути, все сводится к получению файловых дескрипторов, состояние которых изменилось, в бесконечном цикле. То есть тех файловых дескрипторов, которые готовы либо к записи, либо к чтению.

Он работает, следуя набору базовых шагов:

1. Сначала вы регистрируете интерес к событиям ввода-вывода для каждого файлового дескриптора с помощью соответствующего системного вызова. Это часто называют подпиской на событие, которое представляет собой изменение состояния файлового дескриптора. Например, когда клиент подключается к прослушивающему сокету, состояние сокета меняется: в нем появляется запрос на подключение, ожидающий чтения и принятия приложением.

2. Затем вы ожидаете событий ввода-вывода на зарегистрированных файловых дескрипторах, блокируя системный вызов, который зависит от вашего механизма мультиплексирования. Для механизма epoll это процедура `epoll_wait()`. Это может быть либо бесконечное ожидание до тех пор, пока не произойдет хотя бы одно событие, либо может быть указан тайм-аут.

3. Как только блокирующий системный вызов из предыдущего шага возвращает результат, вы получаете произошедшие события.

4. Затем вы определяете тип произошедшего события (например, чтение, запись) и обрабатываете его должным образом, выполняя соответствующую функцию обратного вызова (callback).

5. Наконец, вы повторяете эти шаги, чтобы продолжать асинхронно отслеживать состояние файловых дескрипторов.


### Прослушивающие сокеты: взгляд пользователя

Когда Redis запускается, он создает прослушивающие сокеты. Они обрабатывают входящие запросы на подключение. Redis может создавать TCP-сокеты при подключении по TCP, а также доменные сокеты Unix при локальном подключении. Каждый сокет имеет ассоциированную память, выделенную ядром Linux. Эту память часто называют обобщенным термином "буфер". Это зонтичный термин для всех видов структур данных, используемых для хранения TCP-запроса до его обработки. Обычно все они демонстрируют поведение, подобное очереди. В случае запросов на подключение они принимаются из очереди принятия (accept queue):

Когда Redis принимает запрос на подключение, создается другой сокет. Его обычно называют сокетом подключения (connection socket):

С этого момента он обслуживает все клиентские запросы. Когда прослушивающие сокеты созданы, Redis помещает их в набор epoll для событий чтения:

_Существует N TCP-сокетов для взаимодействия без TLS, N TCP-сокетов для взаимодействия с TLS (тот же адрес, но другой порт) и один доменный сокет Unix в списке интересов экземпляра epoll_

Таким образом, Redis проверяет наличие новых запросов на подключение на каждой итерации цикла событий. Если они есть, Redis запускает определенную процедуру, которую я рассмотрю в ближайшее время.

## Redis принимает запрос на подключение

После того как запрос на подключение оказывается в очереди принятия прослушивающего сокета, он готов к тому, чтобы Redis его принял.

На следующей итерации цикла событий Redis проверяет, доступны ли какие-либо из прослушивающих сокетов для чтения. Если да, то Redis выполняет соответствующую процедуру. Для чтения прослушивающих сокетов, то есть принятия запроса на подключение, это `connSocketAcceptHandler()`. Сначала она создает подключенный сокет, затем создает структуры данных `connection` и `client`, связанные с этим сокетом, и помещает сокет в набор epoll для событий чтения. Эта процедура принимает до 1000 запросов на подключение из очереди принятия на каждой итерации цикла событий.

### Принятие запроса на подключение на сетевом уровне

Эта логика реализована процедурой `anetTcpAccept()` из модуля `anet`. В системах Linux специфическим системным вызовом, используемым для принятия соединений, является `accept4()`. Он похож на `accept()`, но с дополнительным аргументом флага, который позволяет устанавливать различные опции. Redis использует его для автоматической установки принятого сокета в неблокирующий режим.

Вкратце, вот что происходит в ядре. Точкой входа для принятия запроса на подключение является процедура `inet_accept`. Вся тяжелая работа выполняется в специфичной для протокола процедуре `accept()`, которая в случае TCP IPv4 проксирует вызов к `inet_csk_accept()`. Она извлекает элемент из начала очереди принятия и возвращает его.

### Создание структуры данных соединения

Именно это делает `connCreateAcceptedSocket()`. Она устанавливает тип соединения в `CT_Socket`. Это одна из реализаций типа `ConnectionType`, которая содержит множество процедур, связанных с соединениями. Кроме того, недавно созданная структура соединения имеет дескриптор подключенного сокета и состояние `CONN_STATE_ACCEPTING`:

Почему оно все еще `accepting` (принимается)? Несмотря на то, что соединение принято на сетевом уровне, оно еще не готово обслуживать запросы клиента, поскольку его подключенный сокет еще не зарегистрирован в экземпляре epoll.

### Создание структуры клиента

Если общее количество соединений не превышает значение настройки `maxclients`, создается структура клиента. Помимо инициализации всех ее полей, выполняются некоторые операции с подключенным сокетом.

**Алгоритм Нейгла (Nagle's algorithm)** При передаче данных по TCP каждый пакет имеет определенный объем накладных расходов, включая метаданные, такие как заголовок TCP, заголовок IP и, вероятно, заголовок Ethernet. Эти метаданные необходимы для правильной маршрутизации и обработки пакета принимающей стороной.

Например, если полезная нагрузка составляет всего один байт, общий размер пакета будет намного больше одного байта из-за накладных расходов. Если предположить, что размер заголовка Ethernet составляет 14 байт, заголовка IP — 20 байт, а заголовка TCP — 20 байт (это минимально возможный размер), общий размер пакета составит 55 байт. Это означает, что отправка множества маленьких пакетов может привести к значительным накладным расходам и, как следствие, к снижению производительности сети. Чтобы смягчить это, можно использовать алгоритм Нейгла.

Он работает следующим образом. Перед отправкой нового сегмента TCP стек TCP проверяет, есть ли неподтвержденные данные. Если они есть, стек TCP ожидает выполнения одного из следующих условий:

1. Накоплено достаточно данных для заполнения TCP-сегмента. Максимальный объем данных, который может быть отправлен в одном сегменте TCP, называется максимальным размером сегмента (Maximum Segment Size, MSS):


_Алгоритм Нейгла: сценарий, когда отправляющий хост ждет, пока буфер TCP не заполнит весь TCP-пакет_

2. Либо не осталось неподтвержденных данных. Это означает, что все ранее отправленные данные были подтверждены получателем.


_Алгоритм Нейгла: сценарий, когда отправляющий хост ждет, пока все отправленные данные не будут подтверждены (ack'ed)_

Алгоритм Нейгла вызывает задержки при передаче небольших пакетов, когда есть неподтвержденные данные в пути. Таким образом, он может сократить общее количество передаваемых пакетов, что, в свою очередь, может уменьшить перегрузку сети и повысить производительность. При первоначальном установлении соединения этот алгоритм включен по умолчанию.

Для ясности: когда приложение отправляет TCP-запрос и нет неподтвержденных данных, TCP-пакеты отправляются сразу же, даже если их полезная нагрузка составляет всего один байт:

Точно так же, если каждый сегмент TCP заполнен до максимального размера, отправляющая сторона не ждет подтверждения ранее отправленных данных:

Однако в некоторых случаях алгоритм Нейгла может наносить ущерб производительности. Например, в ситуациях, требующих связи в реальном времени или с малой задержкой, буферизация и задержка, вызванные алгоритмом Нейгла, могут быть неприемлемы. В таких случаях для достижения оптимальной производительности может потребоваться отключение алгоритма Нейгла и, следовательно, «синхронное» ожидание подтверждений пакетов.

**Механизм отложенного подтверждения (Delayed acknowledgment)** Отложенное подтверждение (Delayed ACK) — это функция в TCP, которая позволяет получателю отложить отправку пакетов подтверждения (ACK) для полученных данных. Вместо того чтобы немедленно отправлять ACK для каждого отдельного пакета, получатель может дождаться одного из следующих событий:

1. Объединение ACK: принимающий хост может подождать, чтобы накопить несколько ACK перед их совместной отправкой в одном пакете:


2. Piggybacking ACK: хост может отложить отправку ACK до тех пор, пока не сгенерирует пакет с ответом. ACK может быть включен в ответный пакет, что уменьшает количество отдельных пакетов ACK:


3. Тайм-аут: если никакой другой пакет или ответ приложения не инициирует ACK, хост в конечном итоге может отправить ACK по истечении определенного периода тайм-аута. Этот тайм-аут обычно составляет около 200–500 миллисекунд.


Цель отложенного подтверждения — минимизировать накладные расходы сети и повысить эффективность. Задерживая передачу ACK, получатель может уменьшить количество отправляемых мелких управляющих пакетов. Это может быть полезно в сценариях, где происходит обмен большим объемом мелких пакетов данных.

Если на отправляющем хосте включен алгоритм Нейгла (что по умолчанию так), а на принимающем хосте включена функция отложенных подтверждений, два хоста могут ждать друг друга. Эта ситуация фактически является взаимоблокировкой (deadlock), которая может быть разрешена только по тайм-ауту:

_Взаимоблокировка: отправляющий хост ждет ACK для первого пакета, а принимающий хост ждет второго пакета_

**Как Redis решает эту проблему** Redis отключает алгоритм Нейгла, устанавливая опцию `TCP_NODELAY` для подключенного сокета. Другими словами, Redis приказывает операционной системе отправлять ответ клиенту, как только он становится доступным, независимо от его размера. Но Redis сохраняет функцию отложенного подтверждения TCP по умолчанию. Это можно устранить с помощью опции `TCP_QUICKACK`, но Redis ее не использует.

Отключение алгоритма Нейгла на стороне Redis не влияет на поведение клиента. Если `TCP_NODELAY` установлен только на стороне сервера, клиенты будут продолжать буферизовать запросы. Поэтому вам следует рассмотреть возможность установки этой опции и на стороне клиента:

**TCP keepalive** Если указан параметр `tcp-keepalive`, механизм поддержания активности (keepalive) включается в Redis. TCP использует его, чтобы определить, жива ли другая сторона соединения и доступна ли она. Он делает это путем периодической отправки небольших пакетов на другой конец и ожидания ответа. Если соединение считается разорванным, ОС закрывает соответствующий сокет.

Для включения keepalive на сокете должна быть установлена опция `SO_KEEPALIVE`. Частота и количество отправляемых сообщений управляются опциями `TCP_KEEPIDLE`, `TCP_KEEPINTVL` и `TCP_KEEPCNT`. `TCP_KEEPIDLE` указывает время в секундах, в течение которого соединение должно простаивать до отправки сообщений keepalive. `TCP_KEEPINTVL` задает интервал в секундах между последующими сообщениями. `TCP_KEEPCNT` определяет количество сообщений, которые могут быть отправлены до того, как соединение будет признано разорванным.

**Регистрация подключенного сокета в экземпляре epoll** Наконец, подключенный сокет регистрируется для событий чтения в экземпляре epoll. Как и любая функциональность, связанная с подключением, процедура реализована в `ConnectionType`. Для TCP-соединения это `connSocketSetReadHandler()`. Она добавляет сокет подключения в набор epoll для отслеживания событий чтения и устанавливает процедуру `readQueryFromClient()`, которая будет выполняться при поступлении новых данных.

Краткое повторение процесса запуска Redis

Экземпляр Epoll

Экземпляр epoll — это фундаментальная структура данных внутри ядра Linux, включающая в себя два важнейших компонента: список интересов (interest list) и список готовности (ready list). При добавлении файлового дескриптора в список интересов, также известный как набор epoll (epoll set), экземпляр epoll непрерывно отслеживает его асинхронно на предмет событий. События означают изменения в состоянии файлового дескриптора, которые в первую очередь включают в себя два различных изменения.

Первое изменение происходит, когда файловый дескриптор становится доступным для чтения, что указывает на наличие данных. По сути, файловый дескриптор переходит из состояния неготовности в состояние, готовое для чтения. Второе изменение происходит, когда в буфере файлового дескриптора появляется достаточно места для записи данных. В таких сценариях эти файловые дескрипторы обычно называют доступными для записи. Всякий раз, когда происходит любое из этих событий, экземпляр epoll немедленно добавляет соответствующий файловый дескриптор в список готовности:

> Состояние файлового дескриптора, когда он только что изменился. Например, теперь он доступен для чтения или записи.

Для извлечения файловых дескрипторов, которые в данный момент готовы к операциям ввода/вывода, может быть вызвана процедура `epoll_wait()`.

Epoll представляет собой лишь один из нескольких механизмов мультиплексирования, используемых для асинхронного отслеживания состояния файловых дескрипторов. В контексте Redis, когда он собирается на операционной системе Linux, epoll служит выбранной реализацией.

Сокет подключения

Во время запуска Redis создает прослушивающие сокеты (listening sockets) и регистрирует их в экземпляре epoll для событий чтения. Эти прослушивающие сокеты сохраняют все входящие запросы клиентов на подключение в связанном буфере TCP, называемом очередью приема (accept queue):

Redis принимает до одной тысячи запросов на подключение за одну итерацию цикла событий. В результате каждый клиент имеет свой собственный сокет подключения — тот, с которым установлено TCP-соединение. Этот сокет подключения принимает команды своего клиента:

Каждый подключенный сокет также регистрируется в экземпляре epoll для событий чтения. Когда клиент отправляет команду, и она помещается в очередь приема сокета подключения, это означает, что сокет только что стал доступен для чтения, и выполняется `readQueryFromClient()`.

Цикл событий (Event loop)

Цикл событий представляет собой программную конструкцию, которая надстраивается над системным механизмом мультиплексирования. Его суть заключается в бесконечном цикле, который непрерывно извлекает файловые дескрипторы, чье состояние претерпело изменения, указывая на их готовность к операциям чтения или записи.

Работа цикла событий состоит из серии фундаментальных шагов:

1. Первоначально вы регистрируете интерес к событиям ввода-вывода (I/O) для каждого релевантного файлового дескриптора. В случае использования epoll в качестве механизма мультиплексирования — это системный вызов `epoll_ctl()`. Это действие, часто называемое подпиской на событие, означает обнаружение изменения в состоянии файлового дескриптора. Например, когда клиент отправляет запрос, и он принимается сокетом подключения, состояние сокета изменяется, сигнализируя о наличии вновь прибывших данных, ожидающих прочтения приложением.

2. Впоследствии вы ожидаете событий I/O на зарегистрированных файловых дескрипторах путем вызова блокирующего системного вызова, специфичного для выбранного механизма мультиплексирования. В случае механизма epoll это включает использование процедуры `epoll_wait()`. Блокирующий системный вызов может либо ждать неограниченно долго до тех пор, пока не произойдет хотя бы одно событие, либо указывать продолжительность тайм-аута.

3. Как только блокирующий системный вызов из предыдущего шага возвращает управление, вы получаете произошедшие события.

4. Вслед за этим вы определяете тип события, такой как событие чтения или записи, и обрабатываете его соответствующим образом, выполняя связанную функцию обратного вызова, обычно называемую обработчиком (handler) в Redis.

5. Наконец, вы непрерывно повторяете эти шаги, постоянно отслеживая и проверяя состояние файловых дескрипторов асинхронно.


Придерживаясь этого паттерна цикла событий, разработчики могут эффективно управлять операциями I/O и оперативно реагировать на изменения состояний файловых дескрипторов.

Redis получает команду

Именно это делает `readQueryFromClient()`. Сначала проверяется, должна ли команда быть распарсена немедленно или ее следует отложить до тех пор, пока это не сделают потоки IO. Я рассмотрю сценарий с потоками IO в следующей статье, так что здесь у нас этот механизм отключен. Затем Redis читает сырые данные команды из сокета и сохраняет их в буфере запросов клиента. Это часть функциональности, связанной с подключением, и в таких случаях она реализована как указатель на функцию в структуре `ConnectionType`.

Вот пример того, как могут выглядеть сырые данные команды, если клиент отправил `set a 12345`:

```
*3 // это количество аргументов 
$3 // это длина следующей лексемы — в данном случае команды 
set // это сама команда 
$1 // это длина первого аргумента в байтах 
a
$5 // это длина второго аргумента в байтах [cite: 54, 55]
12345; 
```

Это формат мульти-объемного (multi-bulk) запроса — часть RESP, протокола Redis по умолчанию.

Затем Redis читает буфер клиента и парсит команду. Вся тяжелая работа реализована в процедуре `processMultibulkBuffer()`. Она заполняет поле `argv` клиента распарсенными аргументами. Например, для предыдущей команды их три: `set`, `a`, `12345`.

Обратите внимание, что первая строка multi-bulk команды — это всегда количество аргументов. Таким образом, Redis всегда знает, была ли распарсена команда целиком. Это может быть не так, когда, например, текущая команда очень велика и не может поместиться в одном TCP-сегменте. Или когда клиент отправляет несколько команд, и последняя, опять же, не помещается в TCP-сегмент. И если `epoll_wait()` возвращает результат прямо перед приходом второго TCP-сегмента, то Redis получает частичную команду.

Однако всё в порядке: TCP-стек операционной системы подождет, пока он не прибудет, и Redis узнает об этом, вызвав `epoll_wait()` на последующей итерации цикла событий.

Если распарсена вся команда, тогда Redis готов ее выполнить. Именно это делает `processCommandAndResetClient()`. Помимо непосредственного выполнения команды, эта функция выполняет некоторые вспомогательные задачи по учету: сбрасывает поля клиента, используемые для отслеживания парсинга команд, и обновляет некоторую статистику.

Я опишу процесс выполнения команды в одной из своих следующих статей.

Redis записывает ответ

Отправка ответа клиенту состоит из трех логических частей:

1. Redis добавляет клиента в список `server.clients_pending_write`.

2. Затем он формирует ответ в определенном формате.

3. Этот ответ сохраняется в структуре клиента.

4. На последующей итерации цикла событий каждый ожидающий ответ асинхронно записывается в сокет подключения.


Теперь давайте рассмотрим каждую часть более подробно.

Формирование ответа Redis

Форматирование ответа реализовано в наборе процедур `addReply*`, расположенных в файле `networking.c`. Например, это может быть процедура `addReplyBulk`. Сначала добавляется длина ответа. За ней следует тело ответа, и наконец — переход на новую строку.

Например, если ключ `a` равен `12345`, то когда клиент выполняет команду `get` для него, ответ выглядит следующим образом:

```
$5
123456
// это новая строка [cite: 74, 75]
```

Сохранение ответа Redis в клиенте

Внутренне этот ответ может храниться в буфере фиксированного размера и, если он не помещается там целиком, в списке ответов (reply list). Если длина ответа меньше 16 КБ, так что он помещается в буфер ответов, он сохраняется только в поле `buf` структуры клиента. В противном случае оставшаяся часть сохраняется в поле `reply`. Это связанный список. Если в его текущем хвосте есть свободное место, ответ добавляется в его буфер. После этого оставшаяся часть сохраняется в новом элементе списка, который становится новым хвостом. Его минимальный размер составляет 16 КБ, а максимальный размер не ограничен.

Зачем Redis нужен этот механизм? Почему нельзя хранить все в буфере динамического размера? Его размер действительно можно было бы увеличивать и уменьшать с каждым новым ответом. Ну, я не системный разработчик, но могу придумать следующие причины:

1. Перераспределение памяти — недешевая операция, особенно если применять её в масштабах Redis. Например, у вас могут быть тысячи клиентов, получающих ответы примерно в одно и то же время. Это тысячи процедур типа `realloc` за одну итерацию цикла событий. Если дать очень грубую оценку, что одна итерация цикла событий составляет около одной миллисекунды (на деле это может быть на два или даже три порядка меньше), то в сумме это может составить миллионы перераспределений памяти в секунду. Это может повлиять на производительность Redis.

2. Частые перераспределения памяти ведут к её фрагментации. Хотя в Redis есть встроенный способ смягчения этой проблемы, лучше избегать её вовсе.

3. Выделение памяти и последующее её освобождение (`free`) обходится дешевле, чем два вызова перераспределения. Каждое перераспределение включает копирование существующих данных в новый блок памяти другого размера, что может иметь временную сложность O(n), где n — размер перераспределяемых данных. В то же время, процедура `free()` имеет константную временную сложность, обычно O(1). Она лишь освобождает память без каких-либо дополнительных накладных расходов процессора или памяти.


Вот почему Redis использует буфер фиксированного размера в 16 КБ, которого, с одной стороны, достаточно для большинства ответов, а с другой стороны, это не создает слишком больших накладных расходов.

Redis добавляет клиента в список clients_pending_write

Перед установкой ответа Redis убеждается, что клиент находится в списке `server.clients_pending_write`. Такое поведение реализовано в процедуре `prepareClientToWrite()`. За исключением нескольких случаев, Redis проверяет, что у клиента нет сохраненного ответа (если он есть, это означает, что клиент уже находится в списке ожидания записи), и что он в данный момент не находится в потоке IO. Если оба эти условия удовлетворены, выполняется функция `putClientInPendingWriteQueue()`, которая окончательно добавляет клиента, обернутого в `listNode`, в список `server.clients_pending_write`.

Краткое резюме

Думаю, сейчас самое время напомнить вам, как выглядит выполнение команды на более высоком уровне:

> Одиночная итерация цикла событий. Redis парсит команду клиента и формирует ответ.

Сначала принимаются новые события через процедуру `epoll_wait()`. Затем они перебираются в цикле, и для каждого события чтения команда парсится, выполняется, и строится ответ. Обратите внимание, что он еще не отправлен клиенту — это происходит на следующей итерации цикла событий в процедуре `beforeSleep()`. Давайте перейдем прямо к этому.

Ответ Redis отправляется клиенту

Эта логика состоит из двух логических частей. В первой Redis отправляет ответ. Во второй Redis регистрирует сокет подключения для событий записи. Это необходимо в том случае, когда ответ был отправлен только частично. Это может произойти, например, когда сокет заблокирован для записи из-за того, что его буфер отправки переполнен.

Эта логика реализована в процедуре `handleClientsWithPendingWritesUsingThreads()`. Название может ввести вас в заблуждение, заставив подумать, что здесь как-то задействованы потоки IO; на самом деле, это не обязательно так. Механизм потоков IO запускается только в том случае, если это поведение включено в настройках, и если Redis считает всю эту затею целесообразной. Если любое из этих условий не выполняется, ответы доставляются в основном потоке. Эта логика реализована в процедуре `handleClientsWithPendingWrites()`. Её реализация довольно проста. Redis берет всех клиентов с ожидающими записями (как я писал выше, они хранятся в списке `server.clients_pending_write`) и вызывает `writeToClient()` для каждого из них. Как вы можете догадаться, это процедура, которая отправляет ответ клиенту.

**Redis отправляет ответ** Функция `writeToClient` имеет ограничение на то, сколько данных Redis может записать в сокет. Почему? Существуют конкретные сценарии, где вероятность блокировки сокета подключения ниже средней. В таких случаях Redis может потратить слишком много времени на обслуживание ответа одному клиенту. Представьте себе пользователя, который подключается к Redis локально через loopback-интерфейс. Я с трудом могу представить там перегрузку сети. К тому же пропускная способность там чрезвычайно высока.

Другим примером может быть ситуация, когда клиент вообще обходит TCP-стек и подключается к Redis через Unix Domain Sockets. Несмотря на то, что емкость их буфера обычно ниже, записываемые в них данные копируются между процессами. Таким образом, в обоих сценариях блокировка сокетов подключения может занять некоторое время. Следовательно, Redis обслуживает не более `NET_MAX_WRITES_PER_EVENT` байтов для клиента в каждой итерации цикла событий.

Для фактической записи данных используются две процедуры. Обычная `write()` используется для данных, хранящихся в буфере `c->buf`. Это непрерывный блок памяти, и процедура `write` работает именно с этим типом данных.

А вот для списка `reply` используется функция `writev()`. Почему? Потому что связанный список занимает фрагментированную (непрерывную) область памяти. А именно для этого и используется `writev`. Таким образом, Redis перебирает каждый элемент и конструирует массив разрозненных структур `iovec`. Затем он атомарно записывается в сокет подключения. Так что, если бы Redis использовал процедуру `write`, у него было бы два варианта. Первый — сделать несколько вызовов `write`, что дорого: как правило, чем меньше системных вызовов, тем лучше. Второй вариант — скопировать содержимое всех элементов списка `reply` в один непрерывный блок памяти и сделать единственный вызов `write`. Впрочем, я не думаю, что второй вариант уступает подходу с `writev`.

Каждая из этих процедур записи возвращает количество записанных байтов, и если оно равно нулю, `errno` заполняется кодом ошибки, объясняющим, что произошло. Некоторые из них указывают на постоянную ошибку; другие позволяют повторную попытку. Среди последних — `EAGAIN`. Это означает, что буфер отправки сокета подключения полон, и запись блокируется. Иными словами, как гласит комментарий, это означает "Попробуйте еще раз" (Try again).

**Redis подписывается на события записи в сокете подключения** В завершение Redis проверяет, остались ли какие-либо данные, которые еще нужно отправить текущему клиенту. Как я уже упоминал, есть по крайней мере два сценария, когда это может произойти. Первый — достижение предела `NET_MAX_WRITES_PER_EVENT`, а второй — код ошибки, возвращенный процедурой записи, чаще всего `EAGAIN`. В любом случае Redis регистрирует сокет подключения для событий записи. Это означает, что когда он станет доступен для записи, Redis определит это с помощью процедуры `epoll_wait` и отправит оставшуюся часть ответа с ранее установленным обработчиком записи.

Это представлено процедурой `sendReplyToClient`. Она сводится к простому вызову `writeToClient()` — точно такой же процедуры, которая выполнялась при первой попытке в `handleClientsWithPendingWrites`. Единственное отличие состоит в том, что установлен обработчик записи, как следует из второго аргумента. В этом случае, когда все данные в конце концов отправлены, Redis отписывается от событий записи для текущего сокета подключения.

В заключение

Короче говоря, весь процесс выполнения команды выглядит так. На первой итерации цикла событий команда парсится, выполняется, и ответ сохраняется. В процедуре `beforeSleep` следующей итерации Redis отправляет этот ответ клиенту. Если по какой-то причине он отправлен лишь частично, Redis регистрирует сокет подключения текущего клиента на события записи. Когда событие срабатывает, сокет становится доступным для записи, и Redis записывает оставшуюся часть ответа. Этот процесс повторяется до тех пор, пока ответ не будет отправлен целиком. Когда это сделано, Redis отписывается от событий записи.

## Что такое потоки ввода-вывода и когда они могут быть полезны

Если вкратце, потоки ввода-вывода — это обычные потоки, которые параллельно выполняют операции ввода-вывода, а именно: запись ответов и, опционально, разбор команд. Чтобы понять, когда это может быть полезно, давайте сначала рассмотрим два сценария, которые послужили причиной внедрения этой функциональности.

### Проблема: тысячи последовательных записей в сокет данных размером менее 64 кБ в одном основном потоке могут занимать слишком много времени

Предположим, у нас есть экземпляр Redis с одним основным потоком и множеством одновременно подключенных клиентов (представьте тысячи). Затем, примерно в одно и то же время, каждый из них отправляет команду. Redis выполняет эти команды одну за другой, а затем записывает ответ. Как вы, надеюсь, читали в предыдущей статье, эта запись выполняется пакетами размером `NET_MAX_WRITES_PER_EVENT` — то есть по 64 кБ каждый. Напомню, это делается для того, чтобы каждый клиент получил равную долю времени.

Сколько времени займет отправка ответов каждому из них? Допустим, размер каждого ответа составляет ровно `NET_MAX_WRITES_PER_EVENT`. Один вызов `write` с данными такого размера занимает около 20 мкс на моем ноутбуке. Умножив это на десять тысяч, вы получите колоссальные 200 мс. То есть клиент, чья команда была получена последней, столкнется с дополнительной задержкой в 200 мс, вызванной исключительно последовательностью из десяти тысяч системных вызовов `write`.

Примерно столько же времени занимает действительно тяжелый и неоптимизированный поисковый запрос в моем экземпляре PostgreSQL. Даже если я пренебрегу временем обработки данных и передачи по сети, и средняя, и медианная задержки составят примерно 100 мс. В реальности это число было бы в несколько раз больше. Звучит как слишком много для высокопроизводительного хранилища данных в оперативной памяти.

Даже если размер ответа более умеренный, например 1 кБ, длительность вызова `write` не уменьшается пропорционально. Я полагаю, это связано с фиксированной стоимостью самого системного вызова, которая сопоставима со стоимостью перемещения данных такого размера. На моем ноутбуке она составляет около 7–10 мкс. Таким образом, несмотря на некоторый выигрыш в задержке, кажется, что медианная задержка в 35–50 мс — это то, с чем мы можем и должны что-то сделать.

А как насчет чтения и разбора команд? Учитывая, что наиболее частой командой Redis, вероятно, является `GET`, вся команда в формате RESP занимает около 30 байт. Чтение этих данных занимает несколько микросекунд — на самом деле, не сильно отличается от чтения или записи 1 кБ. Таким образом, задержка, вызванная исключительно системными вызовами `read`, аналогична задержке от вызовов `write`.

### Решение: Redis запускает весь ввод-вывод в параллельных потоках

Решение представляет собой классический способ устранения подобных проблем: если в системе есть «узкое место», нужно выявить ограниченный ресурс и масштабироваться горизонтально. «Узкое место» здесь очевидно: это единственный поток, который выполняет все операции чтения и записи. Ограниченным ресурсом в Redis является единственный основной поток, работающий на одном ядре процессора в любой момент времени. Масштабирование подразумевает запуск большего количества потоков на большем количестве ядер ЦП.

Следовательно, это приводит нас к потокам ввода-вывода, которые, как мы надеемся, выполняют все операции ввода-вывода параллельно. На самом деле, именно ОС решает, какой поток запустить на каком ядре ЦП. Но в целом, если общая нагрузка на систему в норме, и ваше приложение использует несколько параллельных потоков, велика вероятность, что они будут работать на разных ядрах. Если же нет, вы всегда можете подправить настройку [CPU affinity](https://github.com/redis/redis/blob/d69109834999c9f06fb31ea4472e0eb3b99d8dc8/redis.conf#L2272-L2273).

Опять же, я говорю об этом конкретном сценарии, когда размер `write` меньше 64 кБ. Теперь давайте рассмотрим следующий сценарий, когда размер записываемых данных выше.

### Проблема: тысячи последовательных записей в сокет данных размером более 64 кБ в одном основном потоке могут занять гораздо больше времени

В этом случае после того, как для каждого клиента записан первый фрагмент размером `NET_MAX_WRITES_PER_EVENT`, Redis подписывается на события записи для каждого сокета соединения. Это означает, что если сеть не перегружена, Redis отправит оставшуюся часть ответа на следующей итерации цикла событий. Другими словами, в лучшем случае, когда сеть не испытывает заторов и буферы отправки быстро освобождаются, Redis отправляет фрагменты размером `NET_MAX_WRITES_PER_EVENT` каждому клиенту на каждой итерации цикла событий до тех пор, пока не будет записан весь ответ:

Даже если размер каждого ответа лишь немного превышает 64 кБ, средняя задержка утраивается. Как я писал в предыдущем разделе, и средняя, и медианная задержки составляют около 100 мс. Если Redis требуется две итерации цикла событий, и средняя, и медианная задержки становятся равными 300 мс. Другими словами, это значение увеличивается на 200 мс на каждой итерации цикла событий. В процентном отношении некоторые клиенты испытывают задержку гораздо хуже, чем другие: клиент, чья команда была получена первой, должен ждать, пока Redis запишет ответы всем остальным 9999 клиентам на первой итерации. Таким образом, ему приходится ждать дополнительные 200 мс. Вы можете почувствовать разницу с предыдущим сценарием, когда дополнительная задержка, вызванная записями, составляла всего лишь его собственный системный вызов `write`, то есть 20 мкс.

### Решение: Redis запускает запись в потоках ввода-вывода

Учитывая, что средний размер команды по-прежнему составляет 30 байт, вклад вызова `read` в общую задержку по-прежнему составляет несколько микросекунд. Если средний размер ответа составляет сотни килобайт, вклад вызовов записи выше. Перенос части чтения в потоки ввода-вывода не будет так заметен, как распараллеливание записи.

### Итак, стоит ли мне попробовать эту функцию?

Как я описал выше, потоки ввода-вывода проявляют себя наилучшим образом, когда клиентов много, а еще лучше — когда ответы велики. Сколько именно «много клиентов» и насколько «ответы велики»? Вы можете провести расчеты для вашего конкретного случая и настроек, но, как правило, гораздо проще включить эту функцию и посмотреть на результат. Но сначала убедитесь, что у вас есть хороший инструмент мониторинга для адекватного сравнения производительности.

И последнее, но не менее важное: обратите внимание, что выполнение команд по-прежнему остается однопоточным! Я покажу это в исходном коде позже, но если вы не дочитаете до того момента, я считаю важным упомянуть об этом факте сейчас.

Давайте рассмотрим, что происходит внутри Redis, когда несколько клиентов отправляют команды. Механика потоков ввода-вывода разворачивается постепенно, поэтому я опишу конкретный пример, начиная с запуска Redis и заканчивая несколькими итерациями цикла событий.

## Конфигурация и запуск Redis

Я предполагаю, что Redis настроен с одним потоком ввода-вывода (`io-threads` равно 2), и чтение вместе с разбором также выполняется в потоках ввода-вывода. Таким образом, наш единственный поток ввода-вывода создается и приостанавливается перед запуском первой итерации цикла событий.

Вот краткое описание того, как создаются и запускаются потоки ввода-вывода.

### Резюме запуска потоков ввода-вывода

Напомню, потоки ввода-вывода запускаются при старте Redis. Если количество потоков, указанное в конфиге, равно `1`, то механизм потоков ввода-вывода фактически отключается. Если их количество не превышает `IO_THREADS_MAX_NUM`, Redis повторяет следующую логику `IO_THREADS_MAX_NUM - 1` раз (исключая основной поток).

Сначала он инициализирует и захватывает мьютекс текущего потока. Затем устанавливает количество клиентов, подлежащих обработке этим потоком, в ноль. И после этого Redis фактически создает поток. `IOThreadMain` — это процедура, которая выполняется в каждом из этих потоков. Это бесконечный цикл, который по сути делает две вещи.

Во-первых, после создания потока ввода-вывода он пытается захватить блокировку, если за ним не закреплено ни одного клиента. Если основной поток уже захватил эту блокировку, то поток ввода-вывода «повисает». И это именно то, что происходит при запуске Redis. Заметьте, однако, что это не активное ожидание. То есть процессор не нагружается. Существует также хитрый цикл `for` на 1 000 000 итераций, который прерывается, когда `getIOPendingCount(id) != 0`. Я вернусь к этому чуть позже.

Второе — это итерация по списку клиентов, назначенных текущему потоку ввода-вывода. Для каждого клиента Redis либо считывает команду, либо записывает ответ. Конкретная стратегия определяется глобальной переменной `io_threads_op`. По завершении список клиентов очищается.

## Первая итерация: Один клиент отправляет одну команду

Я подробно описывал этот случай в предыдущей статье. Вот краткое резюме:

Для начала Redis получает новые события с помощью процедуры `epoll_wait()`. Для каждого события чтения считываются данные. Затем Redis разбирает команду, выполняет её и формирует ответ. Кроме того, клиент сохраняется в списке `server.clients_pending_write`. Важно отметить, что ответ не отправляется клиенту немедленно. Вместо этого это происходит во время следующей итерации цикла событий в процедуре `beforeSleep()`.

Клиенты продолжают присылать новые команды. Аналогично предыдущей итерации, каждый из четырех клиентов отправляет команду. После этого он устанавливает контекст для этих потоков. В нашем случае это контекст записи, на что указывает значение `IO_THREADS_OP_WRITE`. Наконец, основной поток устанавливает количество клиентов, которые должны быть обработаны каждым потоком, и это служит сигналом для каждого потока ввода-вывода о том, что они могут прервать миллионный цикл `for` и начать работу.

> Если миллион итераций займет меньше времени, чем требуется основному потоку для назначения клиентов потокам ввода-вывода, ничего страшного не произойдет: мьютекс блокируется, затем тут же разблокируется, и итерация продолжается до тех пор, пока `getIOPendingCount(id) != 0`. Так что этот цикл на миллион — просто способ не возиться со слишком частой блокировкой и разблокировкой.
>
> Я называю это «сигнализацией условной переменной для бедных», но это не высокомерный выпад с моей стороны. Я понимаю, что у разработчиков Redis, вероятно, были очень веские причины реализовать это именно так — я просто о них не знаю.

Как только клиенты назначены потокам, контекст ввода-вывода определен, а «условные переменные» установлены, все потоки начинают записывать ответы. Это часть шаблона параллелизма «fan-out» (разветвление), который Redis использует здесь. После того как основной поток закончит свою часть, он ждет, пока завершатся все остальные потоки ввода-вывода.

Тем временем каждый поток ввода-вывода выполняет свою долю отправки ответов. Когда они заканчивают, их «условная переменная» обнуляется, и поток ввода-вывода переходит в состояние активного ожидания (busy-wait): он ждет следующую порцию клиентов.

Наконец, в основном потоке следует фаза «fan-in» (сведение): Redis проверяет, у каких клиентов все еще остались ожидающие записи. Для тех, у кого они есть, он подписывается на события записи в соответствующем сокете и устанавливает функцию-обработчик `sendReplyToClient`. С точки зрения низкоуровневой реализации, отправка ответа ничем не отличается от однопоточного сценария.

Эта итерация заканчивается так же, как и все остальные: Redis проходит по сработавшим событиям чтения и вызывает `readQueryFromClient`. Но теперь она ведет себя иначе: эти чтения откладываются для дальнейшей обработки. Кроме того, каждый клиент помещается в список `server.clients_pending_read`. ## Четвертая итерация: Redis считывает и разбирает команды в потоках ввода-вывода

Наконец, Redis может считывать и разбирать команды клиентов в потоках ввода-вывода. `handleClientsWithPendingReadsUsingThreads` проверяет `server.io_threads_active` (которая равна `1`) и `server.io_threads_do_reads` (которая также равна `1`). Концептуально следующая логика не отличается от многопоточной записи; используется тот же подход fan-out/fan-in. Давайте рассмотрим её вкратце.

Часть fan-out проходит следующим образом. Сначала основной поток берет список клиентов с ожидающими чтениями, созданный в конце предыдущей итерации, и распределяет их по потокам. Затем он устанавливает контекст ввода-вывода на чтение. Наконец, он устанавливает «условную переменную» в значение, равное количеству клиентов для обработки (ненулевое значение).

Заметьте, что потоки ввода-вывода уже стали активными на предыдущей итерации цикла событий. То есть они не «висят»: их мьютекс уже разблокирован, и они находятся в режиме активного ожидания новых клиентов. Как только `getIOPendingCount(id) != 0`, они начинают свою работу в зависимости от значения `io_threads_op`. В данном случае они вызывают `readQueryFromClient`. Основной поток тоже выполняет свои чтения.

Важно помнить, что процедура `readQueryFromClient` используется в однопоточном контексте для чтения, разбора (в частности, для мульти-балков используется `processMultibulkBuffer`), выполнения команды и формирования ответа. Но в случае контекста потока ввода-вывода выполнение команды не происходит. После того как команда разобрана, клиент помечается как имеющий ожидающую команду, и на этом всё.

Стоит упомянуть, что во время разбора Redis может создавать ответы, указывающие на ошибку протокола. Как и любой ответ, это сводится к процедуре `addReplyProto`, которая сначала вызывает `prepareClientToWrite`. Эта процедура помещает клиента в очередь ожидающих записей, чтобы Redis прошел по ней через несколько мгновений и отправил ответы. Но есть два условия для того, чтобы клиент там оказался: он еще не должен быть в этой очереди (что естественно) и эта процедура должна быть вызвана вне контекста потока.

Обоснование второго условия заключается в том, что разработчики Redis, вероятно, хотели избежать конкуренции за блокировки (lock contention). При добавлении в очередь в многопоточном контексте это должно делаться с блокировкой (поскольку Redis не использует очереди без блокировок). Однако такой подход потенциально мог стать «узким местом». Поэтому Redis просто избегает этой конкурентной модификации, добавляя клиента в очередь записи либо на стадии fan-in.

Часть fan-in наступает, когда пора собирать результаты. После того как все потоки закончат работу, основной поток сбрасывает контекст ввода-вывода, проходит по клиентам, чьи команды были только что разобраны, и выполняет эти команды. Если они предполагают ответ, он создается так же, как и в однопоточном сценарии; в частности, клиент помещается в очередь ожидающих записей. Теперь вспомните конец предыдущего абзаца — если у текущего клиента была ошибка в команде, он не был помещен в эту очередь. Именно этим Redis занимается сейчас.

Эта очередь ожидающих записей обрабатывается в процедуре `handleClientsWithPendingWritesUsingThreads`. Я подробно описал её в предыдущем разделе.

## Пятая итерация: Завершение работы потоков ввода-вывода

Есть один клиент с ожидающим чтением; он обрабатывается в `handleClientsWithPendingReadsUsingThreads`. Поскольку клиент всего один, он будет обработан основным потоком. Его команда разбирается и выполняется, как обычно. Затем этот клиент помещается в очередь ожидающих записей, которая обрабатывается процедурой `handleClientsWithPendingWritesUsingThreads`. И именно здесь Redis проверяет, стоит ли по-прежнему держать потоки ввода-вывода запущенными.

Если клиентов с ожидающими записями меньше, чем `server.io_threads_num * 2`, Redis останавливает многопоточный ввод-вывод. Именно это происходит в нашем случае: один (`listLength(server.clients_pending_write)`) меньше четырех (`server.io_threads_num * 2 = 2 * 2 = 4`). Основной поток просто блокирует мьютекс каждого потока, и всё. Потоки ввода-вывода «повисают», пытаясь получить уже захваченную блокировку. Активное ожидание прекращается.

Заметьте, что если на предыдущей итерации не было отправлено ни одного клиента, у Redis не будет возможности завершить работу потоков ввода-вывода. Они будут находиться в состоянии активного ожидания до тех пор, пока какой-нибудь клиент не пришлет команду, подразумевающую ответ.

---

Redis ищет команду и проверяет, может ли он ее выполнить

Во-первых, Redis ищет команду. Все команды хранятся в словаре `redisServer.commands`, который заполняется во время запуска. Redis просто извлекает команду из этого словаря. Очевидно, что если она не существует, то она не выполняется.

Перед выполнением команды Redis проводит некоторые проверки. Их действительно много. Вот некоторые из них:

- Redis проверяет, передал ли клиент правильное количество аргументов.

- Redis проверяет ACL, чтобы узнать, разрешено ли клиенту запускать эту команду.

- Если есть проблемы с сохранением данных на диск, и команда является командой записи, а отправлена она обычным клиентом, а не мастером, то Redis отклоняет эту команду.

- Учитывая наличие исправных реплик, настраиваются их количество и максимально допустимое отставание. Если их недостаточно, команда записи не может быть выполнена.

- Очевидно, что Redis не может выполнить команду записи от обычных клиентов (не мастеров) на реплике, доступной только для чтения.

- Учитывая, что чтение устаревших данных не разрешено, если команда, не допускающая чтение устаревших данных, запускается на реплике с разорванным соединением с мастером, она отклоняется. Вы можете узнать, допускает ли команда чтение устаревших данных, если у нее есть флаг `CMD_STALE` — например, `echo` или `hello`. Однако у большинства из них его нет.

- Если прямо сейчас загружается файл RDB или AOF и текущая команда несовместима с загрузкой, то она отклоняется. Как всегда, это свойство определяется как флаг — в данном случае `CMD_LOADING`. То же самое относится к асинхронной загрузке, которая доступна только на реплике во время синхронизации с мастером.

- Если реплика хочет выполнить команду чтения, записи или реплицируемую команду (то есть почти любую команду), мастер Redis отклоняет ее.


Помимо этих проверок, Redis выполняет некоторые обязательные процедуры.

---

Обзор структур, используемых Redis для хранения данных

В каждой базе данных Redis есть два словаря. Первый используется для ключей со сроком действия. Это `redisDb.expires`, и значения, хранящиеся в нем, представляют собой временные метки истечения срока действия. Другой предназначен для значений клиентов; это `redisDb.dict`.

Каждый `dict` в Redis имеет две хеш-таблицы. Обе реализованы как обычный массив; каждый слот, или корзина, содержит список элементов — на случай, если хеши нескольких элементов указывают на один и тот же индекс массива (это известно как коллизия хеширования). Таким образом, это классическая реализация хеш-таблицы, ничего особенного.

Но почему их две? Подождите секунду, я опишу это в следующем разделе.

Какие структуры данных содержат хеш-таблицы? Каждая из них хранит связанные списки `dictEntry`. Ключ клиента находится в его поле `key`. В случае структуры `dict` поле `val` хранит структуру `redisObject`, которая указывает на установленное клиентом значение. В большинстве случаев данные, которые создают и которыми манипулируют клиенты, хранятся исключительно там — в указателе `redisObject.ptr`. Словарь `expires` хранит заданные клиентом временные метки истечения срока действия в поле `dictEntry.v.s64` в миллисекундах.

Вот общий вид того, как выглядит словарь: БД Redis имеет несколько структур `dict`. Каждая содержит две хеш-таблицы. Каждый слот содержит список `dictEntry`. Каждый `dictEntry` имеет ключ и значение. Указатель `*val` ссылается на заданное клиентом значение, обернутое в структуру `redisObject`.

---

Краткий обзор процедур обслуживания: изменение размера и перехеширование словарей

Когда количество элементов в хеш-таблице велико, количество коллизий хеширования увеличивается. Например, если в хеш-таблице четыре корзины и четыреста элементов, то каждая корзина содержит список длиной около сотни элементов. Временная сложность поиска эффективно снижается до линейной — `O(n)` вместо того, чтобы оставаться постоянной — `O(1)`.

Именно поэтому Redis отслеживает размер хеш-таблицы: если количество элементов становится слишком большим, хеш-таблица расширяется. Иметь разреженную хеш-таблицу тоже нехорошо: в Redis есть некоторые процедуры, которые выполняют итерацию по хеш-таблице. Если в ней миллион корзин и всего один элемент, большинство тактов ЦП тратится впустую на саму итерацию, а не на выполнение полезных действий. Именно поэтому Redis сжимает разреженные хеш-таблицы.

Но это не алгоритм "на месте" (in-place). Расширение и перехеширование всех элементов за один раз заняло бы много времени. Вот почему Redis делает это постепенно. Сначала он создает новую хеш-таблицу подходящего размера. Затем он начинает постепенно перемещать туда элементы из первой хеш-таблицы. На каждой итерации cron Redis перехеширует элементы, находящиеся в ста последовательных корзинах. Кроме того, при доступе к хеш-таблице или ее изменении перехешируются элементы из одной корзины. Redis запоминает индекс корзины, на которой он остановился; он хранится в `dict.rehashidx`. Гарантируется, что все предыдущие корзины пусты; все последующие, вероятно, содержат некоторые данные.

Вы можете прочитать подробности реализации этих процессов в первой статье этой серии.

---

Redis запускает некоторые регулярные процедуры перед выполнением команды

Вытеснение клиентов

Redis отслеживает, сколько памяти потребляет каждый клиент. Эти клиенты хранятся в массиве `redisServer.client_mem_usage_buckets`. Клиенты группируются в корзины по объему занимаемой ими памяти, и клиенты хранятся в каждой корзине в виде списка. Например, в первой корзине хранятся клиенты, занимающие до 32 КБ. Во второй хранятся клиенты, занимающие до 64 КБ, и так далее: порог памяти удваивается с каждой последующей корзиной. В последней хранятся клиенты, занимающие более 4 ГБ.

Если настроен лимит памяти на то, сколько памяти могут занимать клиенты, Redis запускает процедуру вытеснения. Под вытеснением клиента подразумевается освобождение всей памяти, связанной с клиентом. Redis начинает с последней корзины `redisServer.client_mem_usage_buckets`, где находятся клиенты, потребляющие наибольший объем памяти. Перед вытеснением каждого нового клиента Redis проверяет, укладывается ли общий объем памяти, занятой всеми клиентами, в заданный лимит. Если да, то вытеснение прекращается. Если нет, то вытесняется следующий клиент в текущей корзине. Если в текущей корзине не осталось клиентов, Redis переходит к следующей. В процессе вытеснения клиента освобождается все, что связано с клиентом: прочитанные, но еще не разобранные команды, разобранные команды с аргументами, неотправленные ответы, отслеживаемые ключи, данные, связанные с pubsub. Кроме того, закрывается сокет подключения.

Вытеснение ключей

Вытеснение ключей — это процесс, который запускается, когда Redis занимает больше памяти, чем задано в настроенном лимите, и ему приходится удалять некоторые ключи. Если политика имеет слово `random` в своем названии, в этом нет ничего особенного: Redis просто выбирает случайный ключ в каждой базе данных и удаляет его. Это быстро и просто.

Но для всех остальных политик логика вытеснения ключей более сложна. На более высоком уровне каждая итерация выглядит следующим образом. Redis берет выборку ключей из первой базы данных, затем для каждого элемента вычисляет свойство, называемое "временем простоя" (idle time), и, наконец, вставляет элемент в пул размером 16 в порядке возрастания. Со временем значение "времени простоя" расширилось, но важнейшее свойство осталось: чем больше значение, тем больше элемент подходит для вытеснения. Этот процесс повторяется для каждой базы данных. В конце концов, когда у Redis появляется пул ключей из каждой базы данных, ключ с наибольшим "временем простоя" удаляется.

Не забывайте, что эта логика выполняется в контексте выполнения команды. Redis не стремится достичь лимитов памяти за один раз — это может занять слишком много времени. Вместо этого ключи вытесняются итеративно и в строгих временных рамках. Максимальная продолжительность настраивается параметром `maxmemory-eviction-tenacity`. Если оно меньше или равно десяти, то указанное значение является коэффициентом для 50 мкс. Если оно больше десяти, но меньше 100, каждая последующая продолжительность на 15% больше предыдущей. Значение `maxmemory-eviction-tenacity`, равное 100, фактически отключает ограничение по времени.

Значение по умолчанию — 10, поэтому перед выполнением команды Redis выполняет вытеснение ключей в течение времени, немногим превышающего 0,5 мс.

Теперь давайте рассмотрим одну итерацию вытеснения ключей более подробно. Я начну с заполнения пула для вытеснения. Следующая логика выполняется для каждой базы данных, но ключи вставляются в один пул фиксированного размера.

**Redis получает выборку ключей из текущей базы данных** Вот что делает функция `dictGetSomeKeys()`. Строго говоря, она не берет выборку. В статистике предполагается, что выборка должна быть случайной и заполняться независимо. Вместо этого данная процедура просто берет `maxmemory-samples` (по умолчанию пять) последовательных элементов, начиная с элемента, выбранного случайным образом из заданного словаря.

Какой словарь подвергается выборке? Если в названии политики есть слово `volatile`, это означает, что для вытеснения подходят только ключи с явно заданными метками времени истечения срока действия. В этом случае выборка производится из словаря `expires`. В противном случае выбирается `dict`.

Когда перехеширование не выполняется, реализация довольно проста. Redis просто выбирает случайную корзину в хеш-таблице и выполняет по ней итерацию, попутно собирая элементы. Сначала он перебирает список каждой корзины. Если она пуста или в ней больше нет элементов, Redis переходит к следующей корзине. Когда он собирает `maxmemory-samples` элементов, он останавливается.

Когда выполняется перехеширование, логика немного усложняется. В целом это выглядит следующим образом. Сначала Redis генерирует случайный индекс. Затем он переходит к корзине первой хеш-таблицы, которая находится по этому сгенерированному начальному индексу, и выполняет итерацию всех элементов из списка этой корзины. Затем он переходит к корзине второй таблицы с тем же индексом и получает оттуда все элементы. Затем Redis повторяет это с последующим индексом и так далее — до тех пор, пока не будет собрано необходимое количество элементов. Чтобы проиллюстрировать этот процесс, я хотел бы рассмотреть один нормальный случай и еще два крайних случая более подробно.

_Перехеширование в процессе: словарь расширяется, а случайный индекс больше индекса перехеширования._

Допустим, случайный индекс равен 5. Итерация выглядит так. Вот как выглядит полученный пул.

Теперь давайте перейдем к более редким случаям.

_Перехеширование в процессе: словарь сжимается, а случайный индекс меньше индекса перехеширования, но больше размера второй хеш-таблицы._

Допустим, случайный индекс равен `4`. В этом случае начальный индекс устанавливается равным индексу перехеширования, поскольку все корзины до индекса перехеширования пусты. Во время заполнения пула выполняется итерация только по большей таблице — той, которая сжимается, поскольку начальный индекс больше размера второй хеш-таблицы. В результате в пуле оказывается только четыре элемента — это меньше, чем запрашивалось. Но элементов больше не осталось.

_Перехеширование в процессе: словарь расширяется, а случайный индекс все еще меньше индекса перехеширования._

Допустим, случайный индекс равен `1`. В этом случае Redis перебирает обе хеш-таблицы, но пропускает уже перехешированные корзины из первой хеш-таблицы — они все равно пусты. Вот полученный пул.

После того как пул готов, Redis переходит к следующей части.

**Redis сортирует пул** Затем Redis сортирует пул в порядке возрастания времени простоя. Для разных политик вытеснения ключей "время простоя" означает разные вещи, но у него есть важнейшее свойство: чем оно больше, тем лучшим кандидатом на вытеснение является соответствующий элемент.

_Время простоя в случае политики LRU_ Название "время простоя" было придумано в те времена, когда единственной политикой, реализованной в Redis, была политика "Наименее недавно использовавшиеся" (Last Recently Used). Когда она применяется, лучшим кандидатом на вытеснение действительно является тот, который использовался реже всего. В этом случае `redisObject.lru` хранит `LRU_BITS` (а именно 24) младших битов временной метки, когда соответствующий ключ искался в последний раз.

Это означает, что это значение переполняется (оборачивается) каждые 194 дня. Другими словами, Redis не может отличить время последнего поиска один день назад от 195 дней назад. Однако для большинства рабочих нагрузок я думаю, что это нормально: если у вас есть ключи, которые ищутся так редко, нет проблемы в том, чтобы обратиться к базе данных один или два раза в год в случае их вытеснения.

В случае политики LRU лучшим кандидатом на вытеснение является тот, который использовался наименее недавно. В этом случае "время простоя" означает именно то, чем оно является: временной интервал, в течение которого текущий ключ не искался: `idle = now() - dictEntry.val.lru`

Теперь давайте перейдем к тому, как вычисляется время простоя при использовании LFU.

_Время простоя в случае политики LFU_ В этом случае `redisObject.lru` хранит комбинацию времени последнего поиска ключа, объединенную с относительной частотой использования.

Время последнего поиска оборачивается каждые 45 с лишним дней, что, я полагаю, все равно не проблема. Redis'у это нужно для вычисления частоты использования ключа; я опишу это через несколько мгновений. Сама частота не является точно вычисленным значением, отражающим физическое понятие частоты. Нам это и не нужно. Для нас достаточно, если она просто коррелирует с частотой использования, чтобы ее можно было использовать для сравнения элементов друг с другом. Поскольку это всего 8 бит, 0 соответствует наименьшей частоте, а 255 — наибольшей.

Теперь, как она обновляется? Это трехэтапный процесс. Во-первых, счетчик уменьшается пропорционально времени, в течение которого соответствующий ключ не искался. Это звучит естественно: если ключ искался в последний раз месяц назад, частота его использования низкая. Именно для этого Redis и хранит время последнего поиска. Во-вторых, счетчик увеличивается вероятностно: чем больше счетчик, тем меньше вероятность его увеличения. Почему так? Потому что нам нужно, чтобы счетчик помещался в 8 бит, поэтому мы замедляем скорость его увеличения. В-третьих, берется новая временная метка в минутах, извлекаются 16 младших битов и объединяются с новым счетчиком.

Как вычисляется "время простоя", когда действует эта политика? В случае LFU нам нужна обратная частота, чтобы чем выше частота, тем меньше "время простоя" и тем меньше этот элемент подходил для вытеснения. Верно и обратное: чем ниже частота, тем больше "время простоя" и тем больше этот элемент подходит для вытеснения. Таким образом, Redis извлекает эту частоту и вычитает ее из ее максимально возможного значения, 255: `idle = 255 - LFUDecrAndReturn(o);`

_Время простоя в случае политики volatile-ttl_ При использовании этой политики вытеснения выборка производится из словаря `expires`. Ключи в нем — это обычные заданные клиентом ключи, но значения — это временные метки истечения срока действия. Мы инвертируем их: чем меньше временная метка истечения срока действия, тем лучше она подходит для вытеснения: `idle = ULLONG_MAX - (long)dictGetVal(de);`

**Redis удаляет ключ с наибольшим "временем простоя"** К этому моменту у Redis есть упорядоченный пул элементов. Затем Redis просто выбирает последний элемент — с наибольшим "временем простоя" — и удаляет его.

Вот, собственно, и всё. Теперь Redis готов выполнить команду клиента.

---

Redis выполняет команду

Наконец, Redis запускает команду. Это сводится к вызову поля `redisCommandProc *proc` разобранной команды клиента. Поэтому, если вы хотите посмотреть исходный код какой-либо конкретной команды, перейдите к таблице команд, найдите там интересующую вас команду и кликните на одиннадцатый аргумент. Эта процедура и является реализацией команды, которую вы ищете.

Общий ход выполнения команды

На самом деле все довольно просто. Сначала Redis находит элемент в словаре с помощью одной из процедур, имя которой начинается с `lookupKey…`. Обычно это `lookupKeyWrite()`, `lookupKeyRead()`, `lookupKeyWriteOrReply()` или `lookupKeyReadOrReply()`. Разница между процедурами `read` и `write` заключается в том, что последняя удаляет элемент с истекшим сроком действия даже на реплике, доступной только для чтения. Процедуры `...OrReply` добавляют ответ на случай отсутствия ключа.

После того как элемент найден, Redis выполняет логику, подразумеваемую командой. Это ядро команды. Если оно достаточно сложное, оно часто выделяется в отдельную процедуру. Например, в случае простой команды `get` найденное значение просто оборачивается в ответ и сохраняется в структуре `client` для последующей отправки. В случае команды `zadd`, где Redis вставляет каждый отправленный клиентом элемент в сортированное множество, эта логика инкапсулирована в процедуру `zsetAdd`, и так далее. Просто выберите интересующую вас команду и погружайтесь в ее реализацию!

Поиск ключа

Большинство команд Redis сначала ищут переданный ключ. Давайте рассмотрим это подробнее.

Семейство процедур `lookupKey…` работает на уровне `redisDb`: все они принимают его в качестве первого аргумента. Это довольно высокий уровень абстракции: самой логики поиска здесь нет. Вместо этого она делегируется процедуре `dictFind`, которая работает на уровне `dict`. Эта процедура представляет собой типичную реализацию логики поиска в хеш-таблице с некоторыми особенностями Redis на случай, если выполняется перехеширование. Во-первых, Redis выполняет один шаг перехеширования, а во-вторых, Redis проверяет обе хеш-таблицы на наличие искомого ключа. Кроме того, если перехеширование в процессе, Redis выполняет шаг перехеширования при каждой операции манипуляции с данными.

Когда `dictFind` возвращает `dictEntry` и он существует, Redis проверяет, не истек ли уже срок действия этого элемента. Если срок истек, но команда выполняется на реплике мастером, элементы никогда не считаются просроченными. В противном случае, если только ключ не искался с помощью `lookupKeyWrite()` или `lookupKeyWriteOrReply()`, он считается просроченным, но физически не удаляется. Кроме того, если присутствует флаг `EXPIRE_AVOID_DELETE_EXPIRED`, текущий ключ с истекшим сроком действия также не вытесняется. Во всех остальных случаях ключ с истекшим сроком действия удаляется. В качестве особого случая это включает в себя ситуацию, когда срок действия ключа истек на реплике, доступной только для чтения, но он ищется с помощью семейства процедур `…Write`.

Наконец, если ключ пережил логику истечения срока действия, его время доступа обновляется в зависимости от используемой политики вытеснения. Из этого есть два исключения. Первое — когда клиент явно указал, что он не хочет изменять время доступа. Второе — когда в данный момент работает дочерний процесс. Почему? Когда дочерний процесс создается путем форка (forking), он унаследует доступ к тем же данным, что и родительский процесс.

Это становится возможным благодаря технологии копирования при записи (copy-on-write, CoW), которая представляет собой механизм, реализованный операционной системой. Если родительский или дочерний процесс изменяет общие данные, копируется новая страница памяти, и изменения видны только процессу, который произвел изменение. Это приводит к увеличению использования памяти и потенциальным накладным расходам на производительность. Вот почему Redis избегает выполнения задач обслуживания во время работы дочернего процесса.
